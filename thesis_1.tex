\chapter*{Abstract}
\addcontentsline{toc}{chapter}{Abstract}

Most developers know, at some level, that code quality deteriorates if it is not actively
managed. The tools available to help are fragmented: a security scanner here, a complexity
metric there, a style checker whose warnings most teams quietly learn to ignore. Assembling
a complete picture of a codebase's health from these disconnected sources is tedious, and
the results rarely guide concrete action. Manual code review, the traditional fallback,
is slow, inconsistent, and limited by the expertise and attention of whoever happens
to be reviewing on a given day.

This thesis describes the design, implementation, and evaluation of \textit{IntelliCode
Review}, a web-based platform that consolidates twelve code quality analysis dimensions
into a single, interactive application. The system accepts Python, JavaScript, TypeScript,
and Java source code and returns a quality report covering security vulnerabilities,
cyclomatic and cognitive complexity, bug probability, code clones, refactoring
opportunities, dead code, technical debt, documentation coverage, performance hotspots,
dependency coupling, and readability, all in under two seconds. Four of the twelve
dimensions are driven by trained machine learning models; the remaining eight are
deterministic AST-based analysers that produce reproducible results without probabilistic
uncertainty. The deterministic modules are validated for pipeline correctness but not
for detection accuracy; precision and recall evaluations for those components remain
future work.

The machine learning results are uneven, and the failures are as informative as the
successes. A peer-review audit caught a label-leakage defect in the original complexity
model, which had produced a spurious $R^2 = 1.000$ by using a target analytically
derivable from the input features. The corrected XGBoost complexity predictor reaches
$R^2 = 0.676 \pm 0.063$ and Pearson $r = 0.980$ against SonarQube. Complexity ranking
holds in cross-project evaluation, transferring to nine of ten held-out repositories (mean
Spearman $\rho = 0.795 \pm 0.186$). The one failure is \texttt{tiangolo/fastapi}, a
decorator-heavy, Pydantic-style codebase outside the training distribution.

The security ensemble (Random Forest + 1D CNN) reaches in-distribution AUC
$= 0.963 \pm 0.012$ on 4,286 labelled samples. Cross-project evaluation collapses to
AUC $= 0.494 \pm 0.189$, essentially at chance. The training data over-represents CVE patterns from
a narrow set of vulnerability categories; repositories with different exposure profiles
produce feature distributions outside that training envelope. High in-distribution AUC
is not evidence of transferability, and this result is the clearest demonstration of that.

The bug predictor reaches AUC $= 0.676 \pm 0.038$ on a standard random split, consistent with the JIT defect prediction literature. A strict temporal split (train
on the earliest 80\% of commits, test on the most recent 20\%) gives AUC $= 0.460$,
below chance. Static file-level metrics do not capture how a codebase evolves; training
on commits that post-date a bug fix produces temporal leakage that inflates the random-split
figure. Under a forward-looking evaluation, the module provides no reliable bug forecast.

The pattern classifier, after removing 14 leaky features that directly encoded the label
assignment rules, achieves a debiased CV F1 of $0.657 \pm 0.016$. The original 26-feature
model's F1 of 0.804 included $+0.082$ of circular label supervision (Wilcoxon $p=0.031$);
that gap is now documented and the debiased model is the production checkpoint.

The evaluation protocol may be the most potentially transferable contribution. Multi-seed variance,
LOPO benchmarks, temporal splits, residual conformal intervals, and a SonarQube
calibration together expose generalisation limits that a single held-out test split would
not catch. A lightweight peer-based evaluation with four graduate students provided
preliminary usability feedback; a controlled study with professional developers remains
future work. The prototype needs no infrastructure beyond a local Python environment and
a browser.

\section{Problem Statement}
\label{sec:problem}

Software development teams have always understood, at some level, that code quality
matters. Defects caught in review are cheaper to fix than defects caught in production;
security vulnerabilities discovered during development cost a fraction of what they
cost after a breach. The evidence supporting these intuitions is extensive. The National
Institute of Standards and Technology estimated software defect costs at approximately
\$60 billion per year in the US economy alone~\cite{nist2002}, and subsequent studies
in large industrial codebases have found consistently that defect density correlates
with code complexity measures like cyclomatic and cognitive complexity~\cite{mccabe1976}.
The knowledge exists. The difficulty lies in applying it at the pace and scale that
modern software development demands.

Manual code review, where a human peer reads changes and comments before they merge,
remains the industry-standard quality gate. It is valuable, but it has
well-documented structural weaknesses. Review quality is inconsistent: the same patch
may receive a thorough critique one day and a perfunctory approval the next, depending
on reviewer load, expertise, and attention~\cite{bacchelli2013, mcintosh2016}.
Reviewers tend to be better at catching style violations than at detecting logical bugs
or security vulnerabilities, particularly in unfamiliar parts of the codebase~\cite{fagan1976}.
As teams grow and deployment cadences accelerate, the reviewer-to-change-set ratio
worsens, and the review process becomes a bottleneck rather than a safety net.

Automated static analysis tools (Pylint, Flake8, Bandit, and their counterparts in other language ecosystems) address some of this gap by running fast, deterministic
checks that do not depend on human attention. They catch obvious issues: undeclared
variables, imports that shadow builtins, string-formatted SQL queries. But they are
fundamentally limited by their architecture. They match code against a fixed rule set;
they do not adapt to the patterns that have historically caused bugs in a particular
project, cannot estimate the probability that a given commit will introduce a regression,
and do not produce a single integrated quality score that a developer can act on
immediately. The output of running three different static analysis tools is three
separate lists of warnings in three different formats, with no guidance on which to
prioritise.

Machine learning offers a path beyond this limitation. Models trained on commit
histories can learn which code patterns precede bug-fixing commits. Models trained on
CVE-annotated code can generalise vulnerability patterns beyond what any hand-written
rule captures. Regression models can predict cognitive complexity from structural metrics
without a human defining the exact formula. The challenge is assembling these
capabilities into a tool that is usable by an individual developer without requiring
infrastructure, configuration, or expertise beyond knowing how to run a web server.

\section{Proposed Solution}
\label{sec:solution}

IntelliCode Review is the response to the challenge stated above: a web-based platform
that consolidates twelve code quality analysis dimensions into a single interactive
application, trained on real open-source code and evaluated against the same cross-project
and temporal standards applied to published defect predictors.

The primary contribution of this thesis is an evaluation protocol for ML-based code
analysis that takes generalisation seriously: multi-seed variance, leave-one-project-out
benchmarks, temporal splits, residual conformal intervals, and a SonarQube calibration
designed to catch leakage rather than confirm it. That protocol is the transferable
result. IntelliCode Review, a twelve-dimension analysis platform backed by those same models, is what makes the evaluation concrete. Without a working system, the
protocol has nothing to evaluate; without the protocol, the system's claims are no
more credible than the optimistic benchmarks in the papers it cites.

The secondary contribution is the system itself: broader dimension coverage in a
self-contained tool than any existing open-source alternative, with an interface
designed around the constraint that findings ignored are findings wasted.

The models are not individually state-of-the-art. The evaluation is the point.

This thesis argues that rigorous evaluation, not model sophistication, is the unsolved problem in applied ML for software engineering. Building the system is the
tractable part; knowing what it actually does in conditions that approximate
deployment is harder and, for a practitioner deciding whether to trust a tool, more
useful.

The design rests on one observation: coverage and speed only matter if
developers actually consult the results. A finding that takes thirty seconds to arrive
gets ignored. A security alert that requires switching tools gets dismissed. The system
was built around that constraint, not around maximising a benchmark metric.

Concretely, the system has six parts. A twelve-model analysis suite covers
security, complexity, bug probability, code patterns, clones, refactoring, dead code,
technical debt, documentation, performance, dependency coupling, and readability.
A FastAPI backend runs all twelve in parallel and returns a full report in under two
seconds. A streaming SSE endpoint pushes partial results to the browser as each model
finishes, so the interface gives immediate feedback rather than a blank screen. A React
single-page application provides a fourteen-tab review view, a Fix Guide with
copy-ready remediation code, a side-by-side Compare page, a pull request review
workflow with optional inline comment posting, and persistent analysis history. A training
pipeline built on real open-source data uses an evaluation protocol designed to expose
generalisation limits, not conceal them. Multi-language support for Python, JavaScript,
TypeScript, and Java uses tree-sitter AST parsers and language-specific security rules.

\section{Research Questions}
\label{sec:rq}

Three questions guide the work:

\begin{description}
  \item[RQ1] Does a unified ML-augmented platform cover a broader set of code
    quality analysis dimensions than traditional single-purpose static analysis
    tools, and do the individual modules produce technically correct outputs on
    verified test cases? The empirical motivation is that developers using fragmented
    tooling systematically under-examine dimensions not covered by their default
    linter~\cite{johnson2013}: a security-only tool produces no complexity signal;
    a complexity-only tool produces no dead-code or clone signal. A unified platform
    removes the decision of which tool to run, which changes what gets examined.
    Coverage is measured by dimension count relative to comparable tools; correctness
    is measured by end-to-end functional testing against hand-labelled inputs.
    Whether broader coverage translates to better developer outcomes is not
    investigated here and remains an open empirical question.
  \item[RQ2] How do the chosen machine learning architectures (XGBoost, RF+CNN
    ensemble, LR+XGBoost ensemble, calibrated Random Forest) perform on each of the
    four trained quality tasks (security, complexity, bug prediction, pattern
    recognition) when trained on real open-source data and evaluated under
    cross-project and temporal protocols?
  \item[RQ3] Can the system be implemented as a practical, self-contained web
    application with response times fast enough for interactive developer workflows,
    without requiring server infrastructure or external service accounts?
\end{description}

\section{Scope and Limitations}
\label{sec:scope}

IntelliCode Review accepts Python, JavaScript, TypeScript, and Java for direct analysis.
Python receives the full twelve-model treatment including ML-based security, complexity,
bug prediction, and pattern recognition. JavaScript, TypeScript, and Java receive
deterministic security scanning (eight vulnerability categories via language-specific
regex rules) and cyclomatic complexity via tree-sitter AST parsers; the trained ML
models do not run for these languages. The repository scanner additionally recognises
twelve further languages (C, C++, Go, Rust, Ruby, PHP, Swift, Kotlin, Scala, R, Dart,
and Shell) for file-level metrics only. The platform integrates with GitHub for repository
scanning: users can connect a personal access token to trigger batch scans of entire
repositories from the Repositories page. The API also accepts optional git metadata
(code churn, author count, file age, past bug count, commit frequency) to augment bug
prediction for Python files.

All four ML models are trained on real open-source code drawn from 33 repositories
across five domains: 4,286 security samples from CVE-annotated commits, 3,015
complexity samples, 8,668 bug-labelled commits, and 6,200+ pattern examples covering
web frameworks, data science libraries, CLI tools, testing infrastructure, and
async networking libraries. Generalisation to unseen production codebases is limited by
that training distribution; model probabilities are risk indicators, not verdicts.

This thesis has a dual identity: it is simultaneously a systems contribution (building
a working multi-model platform) and a methodological contribution (demonstrating what
rigorous evaluation of such systems looks like). These are not in tension. The system
provides the concrete artefact; the evaluation protocol provides the main transferable
insight. Readers primarily interested in the engineering details will find them in
Chapters~\ref{ch:design} and~\ref{ch:demo}; readers primarily interested in the
evaluation methodology will find the core argument in Chapter~\ref{ch:eval}.

\section{Code Review in Software Engineering}
\label{sec:bg-codereview}

Code review is the examination of source code by someone other than the original author,
with the goal of finding defects, improving quality, and spreading knowledge within a
team~\cite{fagan1976}. Fagan's formal inspection process at IBM in the 1970s was the
precursor to modern lightweight peer review. The current dominant form is asynchronous
pull-request review on platforms such as GitHub, GitLab, and Gerrit~\cite{bacchelli2013}.

Empirical studies of review in practice find that reviewers focus on understandability,
coding conventions, and functional correctness~\cite{bacchelli2013}. Security defects
and performance issues get caught far less often~\cite{mcintosh2016}, partly because they demand domain expertise that not every reviewer has and partly because they are harder to spot by reading code than by running it. This is the structural weakness.
That gap, between what manual review
reliably catches and what actually matters for code health, is the problem automated
analysis is meant to close.

\subsection{Pull-request-based review}

Modern teams submit code changes as pull requests, reviewed by peers before merging.
This creates a natural insertion point for automated tooling: the analysis can run on
the PR before any human reviewer opens the diff, filtering out mechanical issues so
that reviewer attention is spent on logic, design, and correctness. IntelliCode Review
is intended for this workflow: a developer submits a file, receives an automated quality report, addresses the findings, and only then asks a peer to review.

\subsection{Cost of Poor Code Quality}

Technical debt, the implied cost of choosing a quick solution over a better one, accumulates with poor code quality. The SQALE (Software Quality
Assessment based on Lifecycle Expectations) methodology~\cite{letouzey2012} quantifies
technical debt in person-hours required to fix each quality issue category. IntelliCode
Review implements a SQALE-inspired debt estimator that aggregates findings from security,
complexity, clone detection, dead code, and refactoring models into a single debt report.

\section{Static Analysis}
\label{sec:bg-static}

Static analysis examines source code without executing it. It encompasses a broad range
of techniques from simple lexical pattern matching to full dataflow and taint analysis.

\subsection{Lexical and Syntactic Analysis}

The simplest form of static analysis scans source code as text, searching for known
dangerous patterns using regular expressions. Tools such as Bandit for Python use this
approach to flag uses of \texttt{eval()}, hardcoded passwords, and insecure deserialization.
IntelliCode Review's security pattern scanner (\texttt{security\_patterns.py}) implements
a curated set of regular expressions covering SQL injection, command injection, path
traversal, hardcoded secrets, weak cryptography, insecure deserialization, code injection,
and XXE vulnerabilities.

\subsection{Abstract Syntax Tree Analysis}

An abstract syntax tree (AST) captures grammatical structure without syntactic noise.
Python's standard library provides the
\texttt{ast} module, which parses source code into an AST and enables tree-walking
visitors. The system leans heavily on AST analysis: the \texttt{ASTExtractor}
class extracts features including function definitions, call sites, import statements,
string literals, class hierarchies, and control flow constructs. Seven of the twelve
models (clone detection, refactoring, dead code, documentation quality, performance,
dependency, and readability) are implemented as AST visitors or pattern matchers.

\subsection{Complexity Metrics}

The field has settled on four main metrics for measuring complexity:

\begin{description}
  \item[Cyclomatic Complexity (McCabe)~\cite{mccabe1976}] Counts the number of linearly
    independent paths through a program's control flow graph. Formally, $V(G) = E - N + 2P$
    where $E$ is edges, $N$ is nodes, and $P$ is connected components.
  \item[Cognitive Complexity~\cite{campbell2018}] A successor metric that penalizes
    nested control structures more heavily, arguing that nesting is a stronger predictor
    of comprehension difficulty than raw path count.
  \item[Halstead Metrics~\cite{halstead1977}] Derive a family of metrics (volume, difficulty,
    effort, predicted bugs) from counts of operators and operands in the source code.
  \item[Maintainability Index~\cite{oman1992}] A composite metric (0--100 scale) combining
    Halstead volume, cyclomatic complexity, and source lines of code, widely used by
    tools such as Visual Studio.
\end{description}

IntelliCode Review computes all four families of metrics in \texttt{code\_metrics.py}.
After correcting a label-leakage defect in the original implementation (see
Section~\ref{sec:eval-methodology}), the XGBoost complexity model uses a 15-dimensional
feature vector with cognitive complexity as the prediction target rather than as a feature.

\section{Machine Learning for Code Analysis}
\label{sec:bg-ml}

\subsection{Defect Prediction}

Defect prediction work has been running since roughly 2000~\cite{nagappan2005, zimmermann2007}.
Early models used product metrics (lines of code, complexity) and process metrics like code churn and developer experience as features for logistic regression or na\"ive Bayes.
Later work showed that random forests~\cite{lessmann2008}, support vector machines, and
neural networks could push accuracy further, though the gains varied widely by dataset.

IntelliCode Review's bug predictor follows this tradition, combining static code metrics
with optional git metadata (code churn, author count, file age, past bug count, commit
frequency) in a Logistic Regression + XGBoost ensemble, where the gradient-boosted tree
component benefits from early stopping regularisation (best iteration at epoch 97).

\subsection{Vulnerability Detection}

ML-based vulnerability detection accelerated after the release of labelled datasets:
Devign~\cite{zhou2019}, D2A, and CodeXGLUE. Approaches range from simple TF-IDF
bag-of-tokens classifiers to graph neural networks over code property
graphs~\cite{yamaguchi2014}, with graph-based methods generally performing better on
tasks that require interprocedural reasoning.

The security component uses a practical two-component ensemble: (1) a Random Forest classifier
operating on a 25-dimensional feature vector (17 code metrics plus 8 security-specific counters
covering dangerous imports, dangerous calls, long string literals, scanner findings at each
severity level, try-block count, string literal count, and call site count), and (2) a 1D
Convolutional Neural Network operating on token ID sequences of length 512.
The ensemble combines both predictions via a weighted average ($w_{RF}=0.55$, $w_{CNN}=0.45$);
these weights were fixed a priori based on validation-set precision-recall balance rather than
optimised by grid search, so they do not introduce an additional tuning degree of freedom.

\subsection{Code Embeddings and Pre-Trained Models}

Pre-trained transformer models for code --- CodeBERT~\cite{feng2020}, GraphCodeBERT~\cite{guo2021}, and CodeT5~\cite{wang2021} --- established strong baselines across vulnerability detection, clone detection, code summarization, and defect prediction.

A CodeBERT-based approach was evaluated for pattern recognition and rejected in favour of a calibrated Random Forest on a 12-dimensional debiased AST feature vector, primarily to avoid the $\sim$500\,MB model download and GPU requirement. After removing 14 leaky features that encoded the label rules directly, the debiased CV F1 is $0.657 \pm 0.016$ (test F1 $= 0.683$, AUC $= 0.846$). Section~\ref{ssec:pattern} details this design.

\subsection{Ensemble Methods}

Ensemble methods combine multiple base learners to reduce variance and improve
generalization. Random forests are a bagging ensemble of decision trees; gradient
boosted trees (XGBoost~\cite{chen2016}) train trees sequentially, each correcting
the errors of the previous. XGBoost handles complexity prediction here, chosen for its strong performance on tabular regression and resistance to
overfitting. The security and bug prediction models both use two-component ensembles
(RF/CNN and LR/XGBoost respectively) to combine complementary representations.

\section{Web Application Architecture}
\label{sec:bg-web}

\subsection{REST APIs and FastAPI}

RESTful APIs are the standard interface for web services, using HTTP verbs
(GET, POST, PUT, DELETE) to operate on resources identified by URIs. FastAPI is a modern
Python web framework built on Starlette and Pydantic that generates OpenAPI documentation
automatically, validates request and response schemas via Python type annotations, and
supports asynchronous request handling via \texttt{asyncio}. FastAPI's performance
(comparable to Node.js and Go for I/O-bound workloads) and type-annotated request
handling made it the right fit for IntelliCode Review's backend.

\subsection{React Single-Page Applications}

React is a component-based JavaScript library for building user interfaces, developed
and maintained by Meta. In React's component model, UI is a function of state: when
state changes, React efficiently re-renders only the affected subtree via a virtual DOM
diffing algorithm. Vite is a modern frontend build tool that provides near-instant
hot module replacement during development and optimised production bundles via Rollup.
shadcn/ui is a collection of accessible, unstyled UI components built on Radix UI
primitives and styled with Tailwind CSS utility classes, giving a consistent design
system without a heavy CSS-in-JS runtime.

\subsection{Browser localStorage}

The Web Storage API provides \texttt{localStorage}, a key-value store that survives
browser sessions with up to 5--10\,MB of storage per origin. IntelliCode Review uses
localStorage for analysis history, user-defined quality rules, repository configurations,
and the GitHub OAuth access token. Storing the token in localStorage rather than
sessionStorage means users do not need to reconnect after closing a tab; a one-time
migration function in \texttt{github.ts} moves any existing token from sessionStorage on
first load, so existing sessions survive the upgrade transparently.
This suits a single-user development tool. A multi-user production deployment would need
a server-side session store and a proper OAuth token management layer.

\section{Automated Code Review Systems}
\label{sec:rel-review}

Automated code review is not a single thing. It covers rule-based linters, statistical
defect predictors, graph neural networks over code property graphs, and commercial
platforms that mix several of these. This chapter covers the work most directly relevant
to IntelliCode Review and explains where this thesis sits relative to each tradition.

\subsection{Traditional Static Analysis Tools}

Pylint~\cite{pylint}, Flake8, and mypy represent the mainstream of Python static analysis.
Pylint checks for coding conventions, error detection, and refactoring suggestions through
a rule-based system. Flake8 wraps pycodestyle, pyflakes, and McCabe complexity into a
lightweight linter. Bandit~\cite{bandit} focuses exclusively on security, scanning for
dangerous function calls, insecure defaults, and known vulnerability patterns using
a plugin architecture.

The limitation is the rule set itself: these tools can only detect what their authors
anticipated. They do not learn from the codebase being analysed, do not predict future
bugs, and produce no integrated score across multiple dimensions. IntelliCode Review
augments rule-based analysis with ML models that capture patterns too context-dependent
to express as fixed rules.

SonarQube~\cite{sonarqube} is a commercial platform offering multi-language static analysis
with quality gates, technical debt tracking, and CI/CD integration. It is the closest
commercial analogue to IntelliCode Review. However, SonarQube uses rule-based analysis
rather than trained ML models for most of its checks, requires significant infrastructure
(a Java server, database), and is primarily oriented toward team-level dashboards rather
than individual developer feedback.

\subsection{CodeClimate and Similar Platforms}

CodeClimate provides automated code review for GitHub repositories, computing a
maintainability score based on duplication, complexity, and code smells. Like SonarQube,
it is rule-based and lacks ML-based defect prediction. IntelliCode Review differs by
incorporating trained models for security, bugs, and complexity, and by running as a
self-contained prototype that requires no external service account or API key.

Amazon CodeGuru Reviewer~\cite{codeguru} uses ML to detect resource leaks, concurrency issues, and security vulnerabilities in Java and Python code through pull request integration. GitHub Advanced Security~\cite{ghadvancedsecurity} (formerly LGTM) applies CodeQL dataflow analysis across multiple languages. DeepScan and Codacy offer ML-assisted analysis as cloud services. All three have access to proprietary training data and production-scale infrastructure that a research prototype cannot match.

IntelliCode Review sits in a different position. It is self-contained, runs without cloud dependencies, and reports cross-project evaluation results — including the failures — rather than presenting only in-distribution benchmarks. The dimension-coverage comparison in Table~\ref{tab:coverage} is most meaningful when read against the open-source tools Bandit, Pylint, Radon, and SonarQube Community Edition, all of which run under comparable infrastructure constraints.

\section{Defect Prediction}
\label{sec:rel-defect}

\subsection{Metric-Based Approaches}

Nagappan and Ball~\cite{nagappan2005} demonstrated that relative code churn metrics predict
defect density with high accuracy across Windows Server 2003 components. Their work
established code churn (lines added, deleted, and changed between revisions) as a key
predictor, which IntelliCode Review incorporates as an optional feature in its bug
predictor. Zimmermann et al.~\cite{zimmermann2007} extended defect prediction to use
network analysis of code dependencies, finding that highly connected components tend to
have more defects.

\subsection{Machine Learning Approaches}

Lessmann et al.~\cite{lessmann2008} conducted a systematic comparison of 22 classifiers
for defect prediction, finding that random forests and support vector machines generally
outperform simpler classifiers. Their benchmark established that no single classifier
consistently dominates, motivating ensemble approaches. IntelliCode Review's use of
LR + XGBoost ensembling follows this reasoning, pairing the linear model's calibrated
probability estimates with XGBoost's ability to capture nonlinear metric interactions.

Kamei et al.~\cite{kamei2013} introduced \textit{just-in-time} defect prediction, which
predicts the defect-proneness of individual code changes (commits) rather than files or
modules. The IntelliCode Review bug predictor borrows the concept of integrating git-level
metadata (churn, author count, commit frequency) with static features.
That design choice also exposes one of the thesis's core findings: under a strict temporal
split, git metadata alone is not enough to overcome distribution shift.

\subsection{Deep Learning for Defect Prediction}

Li et al.~\cite{li2017} proposed CNN-based defect prediction operating on source tokens,
showing that token sequences encode relevant structural information without requiring
manual feature engineering. This directly motivates IntelliCode Review's 1D CNN component
in the security ensemble.

Zhou et al.~\cite{zhou2019} introduced Devign, a graph neural network approach using
joint graph representation (AST, control flow graph, data flow graph, natural language)
for vulnerability detection in C/C++ code. While IntelliCode Review targets Python and
uses simpler representations, the principle of combining structural and lexical views
is reflected in the RF+CNN ensemble.

\section{Code Clone Detection}
\label{sec:rel-clone}

Code clones, duplicate or near-duplicate code fragments, increase maintenance costs by requiring changes to be applied to multiple locations. Roy and Cordy~\cite{roy2007}
defined four types of clones: Type-1 (exact), Type-2 (syntactically identical except for
variable renaming), Type-3 (near-miss, with insertions/deletions), and Type-4 (semantic
clones). The clone detector covers Types 1--3 using normalized AST representation and
TF-IDF cosine similarity between extracted function bodies.

SourcererCC~\cite{sajnani2016} demonstrated that TF-IDF bag-of-tokens with suffix array
indexing could scale clone detection to billion-line codebases. IntelliCode Review uses a
lighter-weight in-memory version appropriate for single-file snippet analysis.
The tradeoff is scale: the approach would not survive a million-line monorepo scan,
but that is outside the tool's stated scope.

\section{Code Readability and Maintainability}
\label{sec:rel-readability}

Buse and Weimer~\cite{buse2010} developed a machine learning model for predicting code
readability based on local features (identifier length, keyword density, line length,
indentation consistency). Their model achieved strong correlation with human readability
judgments. IntelliCode Review's readability scorer implements a multi-dimensional scoring
approach inspired by this work, evaluating naming conventions, comment density, structural
consistency, and cognitive load.

\section{Pre-Trained Models for Code}
\label{sec:rel-pretrained}

Feng et al.~\cite{feng2020} introduced CodeBERT, a bimodal pre-trained model for
programming and natural languages. CodeBERT is pre-trained on six programming languages
from GitHub using masked language modeling and replaced token detection. It achieves
state-of-the-art results on code search, code documentation generation, and code clone
detection. While such models provide strong representational power, they impose substantial
infrastructure requirements (\textasciitilde500\,MB weights, GPU-accelerated inference for
acceptable latency). IntelliCode Review deliberately prioritises deployability over representational power:
the target user is a solo developer running analysis locally on a laptop, and a 500\,MB model download
with a GPU requirement eliminates most of that audience before a single line of code is scanned.
A calibrated Random Forest on 12 debiased structural AST features satisfies the latency budget
(\textless{}200\,ms per file on CPU) without sacrificing practical utility.
The debiased model achieves AUC $= 0.846$ and CV F1 $= 0.657$; the
higher numbers from the original 26-feature model (AUC $= 0.969$, F1 $= 0.804$) were
inflated by label leakage and are not reported as primary results.

\section{Developer tool adoption and alert fatigue}
\label{sec:rel-adoption}

The developer tool adoption literature provides empirical grounding for several design
decisions in IntelliCode Review. Johnson et al.~\cite{johnson2013} surveyed professional
developers about their reasons for not using static analysis tools and found three
dominant barriers: too many false positive warnings, difficulty customising the tool to
their workflow, and a perceived mismatch between warning categories and actual bugs they
cared about.

Christakis and Bird~\cite{christakis2016} extended this analysis with a large-scale
survey at Microsoft, finding that 91\% of developers who stopped using a static analysis
tool cited false positives as the primary reason. Crucially, the threshold was low: even
a 1-in-5 false positive rate led most participants to dismiss the tool's output entirely.
Once a developer has been burned by false alarms on two or three consecutive runs, they
stop opening the results. This \textit{alert fatigue} effect means that precision is not
a nice-to-have property; it determines whether any of a tool's findings get acted on.
IntelliCode Review's design choices follow directly from this evidence: the security model
uses an operating threshold of 0.050 (chosen to favour precision over recall at the cost
of missed detections), findings are grouped by root cause rather than listed by line
number (so a developer sees one SQL injection alert, not twenty), and the bug predictor
explicitly labels its three evaluation-protocol AUC values in the API response to
discourage over-reliance on the raw probability.

The qualitative code review literature adds a complementary perspective. Bacchelli and
Bird~\cite{bacchelli2013} interviewed developers at Microsoft and found that the primary
motivation for code review is finding defects, but the most common outcome is knowledge
transfer and style feedback — a mismatch between intent and practice. Baum et al.\
\cite{baum2017} found that reviewer expertise and familiarity with the codebase predict
review effectiveness more strongly than process variables, suggesting that automated
tools should reduce the cognitive load of first-pass review rather than replace the
expertise-based component. Kononenko et al.~\cite{kononenko2016} surveyed developers at
multiple organisations and identified that unclear or unprioritised findings are the
primary cause of review feedback being ignored — which directly motivates the Fix Guide's
before/after format and effort estimates. Whether IntelliCode Review's findings are
actually acted on in practice is an empirical question this thesis does not answer; a
controlled study with professional developers would be required.

\section{Integrated developer productivity tools}
\label{sec:rel-idt}

GitHub Copilot and Amazon CodeWhisperer address code \textit{generation}: suggesting
completions, functions, and boilerplate. They do not analyse existing code for defects.
IntelliCode Review is in a different position: it does not write code, but scrutinises what is already there.

DeepCode, now absorbed into Snyk Code, uses a machine learning pipeline trained on
GitHub commit histories to detect and fix security vulnerabilities across multiple
languages. It combines learned models with pattern matching, similar in spirit to
IntelliCode Review's RF+CNN security ensemble. The main difference is scope: Snyk Code
is almost entirely security-focused, while IntelliCode Review distributes attention
across twelve quality dimensions, treating security as one signal among many rather than
the primary output. Neither Snyk Code nor DeepCode publishes cross-project or temporal
evaluation results in peer-reviewed venues; independently available benchmarks (Big-Vul,
Devign, D2A) show that commercial tools optimised for low false-positive rates in
production settings generally achieve lower recall on out-of-distribution samples than
in-distribution benchmarks suggest. This evaluation gap is the same problem this thesis
addresses with its LOPO and temporal protocols: in-distribution numbers from the
provider's own benchmark are insufficient to judge deployable performance on a new
codebase. IntelliCode Review's transparency on this point, reporting LOPO AUC = 0.494,
is a differentiator from commercial tools even when those tools achieve better absolute
performance on proprietary internal benchmarks.

\subsection*{Evaluation practice in prior work}

A thread running through the literature reviewed above deserves explicit attention
because it motivates several methodological choices in this thesis. The majority of
defect prediction and vulnerability detection papers report AUC or F1 scores computed
on a single random train-test split. Shepperd et al.~\cite{shepperd2014} and later
Tantithamthavorn et al.~\cite{tantithamthavorn2016} showed that random splits inflate
estimated performance because they allow models to train on future commits relative to
the test set, violating the temporal ordering of software history. The result is that
many published defect prediction AUC values, including several in the papers cited above, are optimistic by a margin that disappears under a strict temporal evaluation.

A related problem affects cross-project evaluation. Most studies evaluate models on
the same project distribution used for training, sometimes drawn from the same
repository. Menzies et al.~\cite{menzies2011} showed that cross-project transfer
often degrades performance substantially; more recent vulnerability detection work
largely confirms this pattern. The issue is not methodological ignorance. Most authors know these limitations. The problem is publication incentive: a paper reporting LOPO
AUC $= 0.49$ on a security model does not get accepted to the same venues as one
reporting AUC $= 0.96$ on a held-out split from the same training set.

This thesis treats temporal leakage and distribution shift as first-class evaluation
results, not footnotes. Every trained component in IntelliCode Review is evaluated
under a temporal split and a leave-one-project-out protocol. The results are reported
whether or not they flatter the models. That choice is the methodological position
this thesis takes, and it is directly motivated by the gap between what the
literature reports and what deployable performance looks like.

PIFF, SAEF, LAMV, and TDCP each address a different category of the reporting gap identified above. Together they form an evaluation protocol that is more demanding than the standard held-out split. No prior work in this survey applies all four.

\subsection*{What the field collectively knows}

The literature surveyed above points to a consensus that the ML for software engineering community has held since roughly 2009 but rarely foregrounds. Cross-project defect prediction reliably degrades below random-split performance: Zimmermann et al.~\cite{zimmermann2007} documented 0\% prediction accuracy on cross-project transfer for some tasks. Random splits overestimate temporal AUC by 10--30\% for static-metric JIT models~\cite{kamei2013}. Label quality, not model capacity, is typically the binding constraint~\cite{shepperd2014}. Every security model that reports in-distribution AUC above 0.90 without a LOPO evaluation is, by that consensus, reporting a figure that does not reflect deployable behaviour.

IntelliCode Review's LOPO results (security AUC $= 0.494$, bug predictor temporal AUC $= 0.460$) are consistent with this prior consensus. They are presented not as failures unique to this platform but as the expected consequence of applying ML to code quality under the constraints that actually apply in practice: limited training data, noisy labels, and distribution shift between training and deployment code.

\section{Motivation}
\label{sec:motivation}

Three problems in current code quality tooling shaped the design of IntelliCode Review.
Each one is a real friction point, not an abstract concern.

\textbf{Fragmentation.} Getting comprehensive Python feedback today means running Pylint
for style, Bandit for security, radon for complexity, vulture for dead code, and possibly
a paid service for technical debt. Each tool has a different configuration format, a
different severity scale, and a different output schema. Putting those results together
takes work that most developers skip, so parts of the picture go unexamined by default.

\textbf{The ceiling on rule-based detection.} Rules catch what their authors anticipated.
Code quality issues are often context-dependent: combinations of features that are
individually harmless can together indicate elevated risk in ways that are hard to
express as fixed patterns. ML models trained on labelled examples of vulnerable or
complex code can capture those distributional patterns, at the cost of requiring
training data and accepting probabilistic outputs.

\textbf{Adoption.} As established in Section~\ref{sec:rel-adoption}, developers abandon tools that produce excessive false positives or demand non-trivial configuration~\cite{johnson2013, christakis2016}. Speed and low friction are not polish; they determine whether analysis results are ever seen.

\section{Design Goals}
\label{sec:goals}

Those three problems translate into five concrete goals that guided the architecture.

\textbf{Comprehensiveness.} The system should cover at least ten distinct quality dimensions (security, complexity, bugs, clones, dead code, documentation, performance, dependency coupling, readability, and technical debt) in one request. No tool-switching, no reconciliation.

\textbf{ML augmentation.} At least three components should use trained models rather
than hand-written rules, capturing distributional patterns that fixed rules cannot
anticipate.

\textbf{Speed.} Full analysis of a 1000-line file within two seconds, with a fast path
returning core results in under 500\,ms for CI pre-commit hooks.

\textbf{Persistence.} Analysis history, custom rules, and repository configurations should
survive browser sessions without a backend database, so a solo developer can use the tool
without any infrastructure setup.

\textbf{Actionability.} Every finding should carry a severity classification, a CWE identifier, and a before/after remediation snippet for common vulnerability patterns.
A finding that identifies a problem without suggesting a fix is only marginally better
than no finding at all.

\section{System Overview}
\label{sec:overview}

Figure~\ref{fig:architecture} shows the high-level architecture of IntelliCode Review,
which is structured across three tiers. The browser-based frontend is a React 18 SPA
where developers submit code, browse results across fourteen analysis tabs, track quality
trends on the Analytics page, and compare two snippets side by side. The browser stores all analysis history in \texttt{localStorage}, with no backend database required.

The Python FastAPI backend receives requests and dispatches them to the appropriate
model classes in parallel. It exposes over twenty endpoints: the full combined analysis
at \texttt{/analyze}, twelve specialised model endpoints, a Server-Sent Events streaming
endpoint at \texttt{/analyze/stream} for progressive results, a fast sub-500\,ms path
at \texttt{/analyze/quick}, and utility endpoints for statistics, feedback collection,
and cache management.

The ML layer consists of twelve model classes. Four of them (security, complexity, bug prediction, and pattern recognition) use trained checkpoints loaded from disk at startup. The remaining eight are deterministic AST-based analysers that produce
reproducible results without probabilistic uncertainty.

repository configurations must persist across browser sessions without a server-side
database. Every finding must be accompanied by a severity classification and a
concrete, human-readable recommendation.

\section{System Architecture}
\label{sec:arch}

IntelliCode Review follows a classical three-tier web architecture with a strict separation
of concerns. The React single-page application running in the browser communicates
with the Python FastAPI backend exclusively over HTTP/JSON; no shared in-memory state
exists between the two tiers. The FastAPI layer in turn invokes a collection of ML model
objects that are loaded once at server startup and held in a singleton
\texttt{ModelRegistry}, so model load time is not incurred on every request. This design decouples the analysis logic entirely from HTTP handling,
so individual models can be tested or replaced without touching the API layer.

The frontend, backend, and ML model layer correspond to the three boxes in
Figure~\ref{fig:architecture}. Within the frontend, all persistence is managed via the
browser's \texttt{localStorage} API; a typed TypeScript service layer (\texttt{api.ts},
\texttt{reviewHistory.ts}, \texttt{auth.ts}) abstracts HTTP calls and storage reads so
that page components contain no direct network or storage logic. Within the backend,
an in-memory hash-keyed cache (maximum 200 entries) is consulted before running
models, avoiding redundant computation on repeated identical submissions.

\section{Feature Extraction Layer}
\label{sec:features}

Feature extraction is the shared foundation on which both the ML models and the
deterministic analysers operate. Several shared modules provide reusable computation
whose outputs are passed downstream to the model classes; the primary ones are described
below.

\subsection{Multi-language adapter}
\label{ssec:multi-language-adapter}

The multi-language adapter (\texttt{features/multi\_lang/adapter.py}) detects the source
language from the filename extension and routes the file to the appropriate parser.
For Python, it delegates to the \texttt{ASTExtractor} and the \texttt{ast} module.
For JavaScript, TypeScript, and Java, it uses tree-sitter grammars
(\texttt{tree-sitter-javascript}, \texttt{tree-sitter-typescript},
\texttt{tree-sitter-java}) to parse a language-specific AST and extract cyclomatic
complexity via control-flow node counting. Language-specific regex rule sets run the
eight security vulnerability categories against the source text of each supported language.
The ML models (trained on Python data) are not invoked for non-Python files;
the adapter returns deterministic security and complexity results instead. This
boundary is explicit rather than silent: the response includes a \texttt{language}
field and a note that ML dimensions are unavailable, which avoids presenting
interpolated predictions for code distributions the models have never seen.
The regex rules for JavaScript, TypeScript, and Java were adapted from the Python
rule set with syntax-appropriate modifications. To provide a preliminary accuracy
estimate for the JavaScript rules, a NIST SARD-inspired evaluation was conducted
using a manually labelled set of 52 JavaScript code snippets (26 vulnerable, 26
clean) covering eight CWE categories: XSS (CWE-79), code injection via
\texttt{eval} (CWE-95), SQL injection (CWE-89), weak cryptography (CWE-338),
prototype pollution (CWE-1321), path traversal (CWE-22), open redirect (CWE-601),
and NoSQL injection (CWE-943). The evaluation script is available at
\texttt{backend/evaluation/js\_security\_eval.py} and results are saved to
\texttt{evaluation/results/js\_security\_eval.json}.

\subsection{AST Extractor}

The \texttt{ASTExtractor} class parses Python source code using the standard
\texttt{ast} module and walks the resulting tree in a single pass, collecting function
and class definitions (names, line ranges, parameter counts, decorator lists), import
statements, call sites, string literals, and control-flow construct counts. The nesting
depth of control-flow blocks is tracked as a running maximum during the traversal.
A secondary tokenisation pathway produces integer token-ID sequences of length 512
(padded or truncated), which serve as input to the 1D CNN security component; the
vocabulary used for this mapping is constructed once during training and persisted
alongside the model checkpoints.

\subsection{Code Metrics}

The \texttt{compute\_all\_metrics()} function computes a 15-dimensional feature vector
shared across the complexity, security, and bug prediction models. The two most
theoretically grounded metrics are cyclomatic complexity and the Maintainability Index:

\begin{align}
  V(G) &= E - N + 2P \quad \text{(cyclomatic complexity)} \label{eq:cc}\\
  MI &= 171 - 5.2 \ln(HV) - 0.23 V(G) - 16.2 \ln(SLOC) \label{eq:mi}
\end{align}

where $HV$ is Halstead volume, $V(G)$ is cyclomatic complexity, and $SLOC$ is source
lines of code.

Cognitive complexity is computed separately and used as the XGBoost complexity model's
prediction target, not as a feature. It increments a counter for each control flow
branching point with additional penalty for each level of nesting:

\begin{equation}
  CC = \sum_{i} (1 + \text{nesting\_depth}(i)) \label{eq:cogcomplexity}
\end{equation}

Excluding cognitive complexity from the 15-dimensional input vector is necessary to
avoid target leakage: a model that can observe its own target in the feature set
learns a trivial identity mapping.

\subsection{Security Pattern Scanner}

The \texttt{scan\_security\_patterns()} function implements a regular-expression-based
vulnerability scanner covering eight vulnerability categories (Table~\ref{tab:vulntypes}).
Each category has associated CWE identifiers and severity classifications (critical, high,
medium, low). The scanner runs on every analysis request and its findings serve both as
direct output and as features for the ML ensemble. Framework-specific taint source
detection is handled by a companion module, \texttt{context\_analyzer.py}, which
identifies user-controlled entry points: Flask route parameters (\texttt{<param>} in
URL rules), FastAPI dependency-injected arguments annotated with \texttt{Query},
\texttt{Path}, \texttt{Body}, or \texttt{Form}, and Django URL captures from
\texttt{re\_path} and \texttt{path} definitions. Recognising these as user-controlled
inputs allows the scanner to prioritise findings that lie on a data-flow path from
an external boundary to a dangerous sink, reducing false positives on internal
utility functions that never receive untrusted data.

\subsection{Security Vulnerability Detection: RF + 1D CNN Ensemble}
\label{ssec:security}

The security detection model (\texttt{security\_detection.py}) uses an ensemble of two
complementary classifiers. The feature vector for the Random Forest has 25 dimensions:
the 17 code metrics from \texttt{code\_metrics.py} plus 8 security-specific counters
(dangerous import count, dangerous call count, long string count, scanner findings at
each severity level, try block count, string literal count, call site count).

The 1D CNN operates on a padded/truncated sequence of 512 integer token IDs. Its
architecture is:

\begin{equation}
  \text{TextCNN}: \text{Embed}_{10000 \to 128} \to
  \underbrace{\text{Conv1d}_{k \in \{3,5,7\}} \to \text{ReLU} \to \text{MaxPool}}_{\times 3}
  \to \text{Concat} \to \text{Dropout}_{0.5} \to \text{Linear}_{768 \to 2}
\end{equation}

The ensemble combines both components:
\begin{equation}
  p_{\text{vuln}} = \frac{w_{RF} \cdot p_{RF} + w_{CNN} \cdot p_{CNN}}{w_{RF} + w_{CNN}},
  \quad w_{RF} = 0.55,\; w_{CNN} = 0.45
\end{equation}

The Random Forest uses 200 trees with Platt scaling for probability calibration
(\texttt{CalibratedClassifierCV}, \texttt{cv=5}, \texttt{method='sigmoid'}).

cognitive complexity itself to prevent target leakage). The predicted score is mapped

\subsection{Bug Prediction: Logistic Regression + XGBoost Ensemble}
\label{ssec:bugs}

The bug predictor (\texttt{bug\_predictor.py}) combines static code features with optional
git metadata. The static feature vector is the same 15-dimensional code metrics vector.
Git metadata (when provided) adds 5 features: code churn, author count, file age in days,
past bug count, and commit frequency.

The Logistic Regression component uses the full 20-dimensional combined vector. The XGBoost
component is a gradient-boosted classifier trained with early stopping (best iteration at
epoch 97, preventing overfitting beyond that point). The ensemble averages LR and XGBoost
probabilities:

\begin{equation}
  p_{\text{bug}} = \frac{p_{LR} + p_{XGB}}{2}
\end{equation}

Training was performed on 3000 real OSS samples labelled via SZZ-style commit heuristics
and Bandit annotations. Held-out AUC scores: LR = 0.677, XGB = 0.710, ensemble = 0.704.
A heuristic fallback is used when checkpoints are unavailable, based on thresholds applied
directly to cyclomatic complexity and Halstead bugs metrics.

Risk levels are classified as:
\begin{itemize}
  \item \textbf{Critical}: $p > 0.75$
  \item \textbf{High}: $0.50 < p \leq 0.75$
  \item \textbf{Medium}: $0.25 < p \leq 0.50$
  \item \textbf{Low}: $p \leq 0.25$
\end{itemize}

\subsection{Pattern Recognition: Calibrated Random Forest}
\label{ssec:pattern}

The pattern recognition model (\texttt{pattern\_recognition.py}) is a Random Forest
classifier with 300 trees trained on a 12-dimensional debiased AST feature vector
extracted per code snippet. The four output classes are: \texttt{clean},
\texttt{code\_smell}, \texttt{anti\_pattern}, and \texttt{style\_violation}. Platt
scaling (\texttt{CalibratedClassifierCV}, \texttt{method='sigmoid'}, \texttt{cv=5})
calibrates the per-class probabilities.

The model was trained on 6,200+ real code samples sourced from 28 OSS repositories
spanning web frameworks, data science libraries, CLI tools, and testing infrastructure,
labelled using a tool-consensus oracle combining PyNose output with metric thresholds.
A label leakage audit (Section~\ref{ssec:audit}) found that the original 26-feature
model trained on CC, SLOC, Halstead, and MI features directly encoded the label
assignment rules as training features, producing circular self-supervision rather than genuine structural pattern detection. The model was retrained on 12 purely structural
AST features: \texttt{n\_functions}, \texttt{n\_classes}, \texttt{n\_try\_blocks},
\texttt{n\_raises}, \texttt{n\_with\_blocks}, \texttt{max\_nesting\_depth},
\texttt{max\_params}, \texttt{avg\_params}, \texttt{n\_decorated\_functions},
\texttt{n\_imports}, \texttt{max\_function\_body\_lines},
\texttt{avg\_function\_body\_lines}. The debiased checkpoint
(\texttt{checkpoints/pattern/rf\_model.pkl}) achieves CV F1 $= 0.657 \pm 0.016$, test
F1 $= 0.683$, accuracy $= 66.5\%$, AUC $= 0.846$. The Random Forest architecture was
chosen over fine-tuned CodeBERT to avoid the $\sim$500\,MB model download and GPU
requirements while still achieving reasonable held-out performance at this corpus size.
Techniques such as LoRA fine-tuning~\cite{hu2022} and ONNX quantisation can reduce
transformer footprints substantially; at the 1,153-sample corpus size used here, the
expected gain over a well-regularised Random Forest is unlikely to justify that
engineering investment, but the trade-off should be revisited when more labelled data
are available.

\subsection{Code Clone Detection: TF-IDF Cosine Similarity}
\label{ssec:clones}

The clone detector (\texttt{code\_clone\_detection.py}) extracts function bodies from the
submitted code, normalizes them (stripping comments, whitespace, and optionally renaming
identifiers for Type-2 detection), and computes pairwise TF-IDF cosine similarity.
Function pairs with similarity above $0.85$ (Type-1), $0.70$ (Type-2), or $0.55$ (Type-3)
thresholds are reported as clones. The result includes clone pairs, their type
classification, and similarity scores. The thresholds are conservative at the Type-3
level to limit false positives at the cost of missing some semantic clones; this is the
correct trade-off for a tool where developer attention is the scarce resource.

\subsection{Refactoring Suggester: AST Rule Engine}
\label{ssec:refactoring}

The refactoring suggester applies a set of threshold-based AST rules derived from
established refactoring catalogues~\cite{fowler1999}. Functions exceeding 40 lines
are flagged for method extraction; nesting depth greater than three levels triggers
a reduce-nesting suggestion; boolean expressions with more than four operators are
candidates for simplification. Three further rules target identifier clarity (overly
abbreviated names), type annotation coverage (unannotated public APIs), and
parameter count (more than five arguments). Each suggestion carries an estimated
remediation effort in minutes and a priority score used to rank the output list. Ranking
by priority rather than by severity class ensures that a developer working through
findings starts with the change that delivers the most measurable improvement per minute
of effort, which distinguishes a usable tool from a warning list.

\subsection{Dead Code Detector: AST Definition/Usage Graph}
\label{ssec:deadcode}

The dead code detector (\texttt{dead\_code\_detector.py}) builds definition and usage
sets by walking the AST. It identifies: (1) imported names that are never referenced,
(2) defined functions or classes that are never called or instantiated, (3) code after
\texttt{return} or \texttt{raise} statements (unreachable code), (4) empty \texttt{except}
blocks that swallow exceptions silently, and (5) variables assigned but never read.
Each finding includes line number, symbol name, and dead code category.

\subsection{Technical Debt Estimator: SQALE-Inspired Aggregator}
\label{ssec:debt}

The technical debt estimator (\texttt{technical\_debt.py}) aggregates findings from all
other models into a debt report following the SQALE methodology~\cite{letouzey2012}.
Debt is estimated in minutes of remediation effort per finding (Table~\ref{tab:debtrates})
and summed into category totals. The overall debt rating (A--E) is determined by the
ratio of total debt to estimated development time.

\subsection{Documentation Quality Analyzer}
\label{ssec:docs}

The documentation analyzer (\texttt{doc\_quality\_analyzer.py}) walks the AST collecting
all function and class definitions and examining their docstrings. It computes:
(1) docstring coverage (fraction of public functions/classes with docstrings),
(2) docstring completeness (presence of parameter, return, and raises sections),
(3) docstring length adequacy (minimum one sentence), and (4) overall quality grade (A--F).
Functions and classes lacking docstrings or having inadequate ones are reported individually.

\subsection{Performance Hotspot Predictor}
\label{ssec:performance}

The performance analyser detects AST patterns that correspond to well-known Python
performance anti-patterns: nested loops (O($n^2$) complexity risk), string
concatenation inside loops (quadratic memory allocation from repeated object
creation), I/O calls inside loops, mutable default arguments (shared state across
invocations), list comprehensions where generator expressions would suffice, and
global variable access in hot paths. Because these patterns are detectable
syntactically without execution, false negative rates are low, though interprocedural
patterns (e.g., a loop in one function calling an I/O-heavy function in another)
remain outside the scope of single-file static analysis.

\subsection{Dependency and Coupling Analyzer}
\label{ssec:dependency}

The dependency analyser constructs a module-level import graph from the AST and
derives three coupling indicators. Fan-out counts the number of distinct imported
modules, providing a rough measure of efferent coupling. Instability is computed as
$I = C_e / (C_e + C_a)$, where $C_e$ is efferent coupling and $C_a$ is afferent
coupling approximated from repeated imports of the same module across the code
under analysis. Wildcard imports (\texttt{from~module~import~*}) are flagged
separately because they pollute the local namespace in a way that makes dead-code
analysis unreliable and is classified as CWE-1104 (use of unmaintained third-party
components). These three indicators are combined into a composite coupling score
(0--100, higher is worse) that surfaces in the dependency tab of the review detail
page.

\subsection{Code Readability Scorer}
\label{ssec:readability}

The readability scorer is inspired by the Buse and Weimer~\cite{buse2010} feature
set but extends it to four distinct sub-dimensions, each contributing up to 25 points
to an overall score of 100. Naming quality rewards adequate identifier length and
consistent \texttt{snake\_case} convention while penalising single-character names
beyond conventional loop counters. Comment quality measures comment density,
the ratio of inline to block comments, and the presence of section-level headers.
Structural quality evaluates function length distribution, control-flow nesting depth,
and blank-line separation. Finally, cognitive load penalises deeply nested boolean
expressions, magic number literals, and excessively long lines. The four sub-scores
are summed and mapped to an A--F grade using the same thresholds as the complexity
model, so the grading language is consistent across all scoring dimensions.

\subsection{Real Dataset Construction}

All training data is sourced from real open-source Python projects rather than synthetic
generation. The decision against synthetic data was deliberate: synthetically generated
vulnerable code often lacks the structural context that makes real-world vulnerabilities
hard to detect, and complexity distributions in synthetic corpora tend to be artificially
uniform.

The \texttt{fetch\_real\_datasets.py} script mines four datasets from 33 repositories
across five domains (web frameworks, data science, CLI tools, testing infrastructure,
async networking). Complexity samples (3,015 functions) include Django, Flask, requests,
pip, NumPy, pandas, scikit-learn, pytest, and urllib3, among others; cognitive complexity
scores are computed by the same AST traversal used in production, giving ground-truth
targets grounded in real distribution. Security samples (4,286 functions) balance clean
code from the same OSS repositories against vulnerable samples from CVE-annotated commits.
Bug samples (8,668 commits) use SZZ-style heuristics to label commits whose message
contains ``fix'', ``bug'', or ``error'' as high-risk. Pattern samples (6,200+ functions)
are assigned one of four class labels using structural thresholds: god class, long method,
feature envy, or clean. The construct-validity caveat (that these thresholds overlap with input features) is documented in Sections~\ref{sec:model-results}
and~\ref{ssec:why-ml-fails}.

\subsection{Model Training}

The master training script \texttt{train\_all.py} runs the four task-specific trainers
in sequence: complexity, security, bug prediction, and pattern recognition. Each trainer
writes a checkpoint and a \texttt{metrics.json} file recording the final evaluation
results. Checkpoints are pickle files for scikit-learn and XGBoost models and PyTorch
state dicts for the 1D CNN.

Two training decisions worth explaining are the security threshold and the Windows
parallelism constraint. The operating threshold of 0.050 for the security classifier
was selected from the precision-recall curve on the expanded 4,286-sample training set
rather than on a separate calibration split. This means the threshold is optimistically
fitted to the training distribution; it may not generalise to codebases with different
vulnerability profiles, and the LOPO AUC of 0.494 is evidence that it does not. For
the pattern classifier, cross-validation is forced to \texttt{n\_jobs=1} on Windows
because Python's \texttt{multiprocessing} module uses \texttt{spawn} semantics instead
of \texttt{fork}, and the calibrated Random Forest contains closures that are not safely
picklable under that scheme. The single-threaded constraint is a known Windows-specific
limitation and has no effect on the quality of the trained model.

A fifth training artefact, the Code Grammar Anomaly Model (CGAM), trains a
Variable-Order Markov Model over depth-first AST node sequences on unlabelled
clean Python code (\texttt{training/train\_cgam.py}). CGAM learns the statistical
grammar of idiomatic Python (which node sequences are common, which are rare) and
assigns an anomaly score to any submitted snippet. The current checkpoint
(vocabulary = 102 node types, trigrams = 7,846, median train perplexity = 4.53)
is stored at \texttt{checkpoints/cgam/vomm.pkl}. CGAM is exposed at
\texttt{POST /analyze/grammar-anomaly} and runs as part of the full
\texttt{POST /analyze} pipeline, returning a \texttt{grammar\_anomaly} field in the
response. It is also included in the model registry at startup (\texttt{ModelRegistry}
loads the checkpoint and reports status on \texttt{GET /models}). The intended use is
out-of-distribution detection: a high anomaly score (low corpus perplexity under the
grammar model) flags code whose AST structure lies outside the training distribution,
which is a useful pre-filter before relying on ML model probabilities. No held-out
precision/recall evaluation of CGAM as a standalone OOD detector has been completed;
its predictive validity as a rejection criterion is left for future work.

\section{Backend API}
\label{sec:api}

The FastAPI application (\texttt{main.py}) defines the model registry as a singleton
class (\texttt{ModelRegistry}) that is populated during the application startup lifespan:

\begin{lstlisting}[caption={FastAPI lifespan: loading ML models at startup},
                   label={lst:lifespan}, style=pythonstyle]
@asynccontextmanager
async def lifespan(app: FastAPI):
    """Load models on startup."""
    print("Loading ML models...")
    try:
        registry.security = EnsembleSecurityModel(
            checkpoint_dir="checkpoints/security")
        print("  [OK] Security detection model")
    except Exception as e:
        registry.load_errors["security"] = str(e)
    # ... (other models loaded similarly)
    registry.clone_detector = CodeCloneDetector()
    registry.refactoring_suggester = RefactoringSuggester()
    # ... (all lightweight models always load)
    yield
    print("Shutting down.")
\end{lstlisting}

The combined analysis endpoint (\texttt{POST /analyze}) accepts a JSON payload with fields
\texttt{code}, \texttt{filename}, \texttt{language}, and optional \texttt{git\_metadata},
and returns a \texttt{FullAnalysisResponse} JSON object containing results from all twelve
models plus an \texttt{overall\_score} (0--100) and a \texttt{status} field (``clean'',
``action\_required'', or ``critical''). Results are cached in a hash-keyed in-memory
dictionary (maximum 200 entries) to avoid re-running expensive models on repeated
identical submissions.

Three additional analysis endpoints extend the core capability:

\begin{itemize}
  \item \texttt{POST /analyze/quick} — runs only security pattern scanning, complexity
    metrics, and readability scoring, returning a lightweight result in under 500\,ms.
    This endpoint is suitable for IDE integrations or CI pre-commit hooks where full
    analysis latency is prohibitive.
  \item \texttt{POST /analyze/stream} — a Server-Sent Events (SSE) endpoint that streams
    partial analysis results as each model completes. Because browser \texttt{EventSource}
    only supports GET requests, the frontend uses the \texttt{fetch} API with a
    \texttt{ReadableStream} reader to consume the SSE response; named events
    (\texttt{security}, \texttt{complexity}, \texttt{bugs}, etc.) update the UI
    progressively as each model finishes.
\end{itemize}

Three utility endpoints support operations: \texttt{GET /stats} returns aggregate
cache statistics used by the Admin page; \texttt{POST /feedback} stores developer
ratings for future model improvement; and \texttt{POST /cache/clear} flushes the
analysis cache on demand. CORS is open to all origins for the local prototype and
should be restricted to the frontend origin in a production deployment.

A separate endpoint group handles GitHub pull request analysis.
\texttt{POST /github/analyze-pr} accepts a PR URL and a personal access token.
The PR URL is validated at the Pydantic schema level using a \texttt{Field(pattern=...)}
regex that rejects any string not matching the canonical GitHub PR URL format
(\texttt{https://github.com/owner/repo/pull/N}), which prevents downstream
failures before any GitHub API calls are made. The endpoint runs in two modes
controlled by a \texttt{post\_comments} boolean flag. When \texttt{false} (the default),
it fetches each changed file, runs the full analysis pipeline, and returns the results
to the browser without touching GitHub. The developer reviews the findings in the
frontend and, if satisfied, calls the same endpoint again with \texttt{post\_comments=true}
to post the results as inline review comments on the changed lines. This two-step design
prevents posting machine-generated comments the developer has not seen, which reduces
noise in the PR review thread and avoids the awkwardness of retracting a false-positive
comment from a public repository.

\section{Frontend Implementation}
\label{sec:frontend}

The frontend is a React~18 single-page application built with Vite and TypeScript,
using React Router~v6 for client-side routing. Ten pages are accessible from the

additional routes (Review Detail, Models, CI integration, Diff viewer) are reachable

details are in Appendix~\ref{app:frontend}.

Three design choices shaped the frontend architecture.
First, the SSE endpoint \texttt{/analyze/stream} populates the Security tab within
200\,ms while remaining models continue running in the background, so the interface
gives immediate feedback rather than a blank screen.
Second, the Fix Guide tab (Tab~14) provides copy-ready remediation code for the
five most common vulnerability types, so a developer who sees a SQL injection finding
can paste the parameterised query template directly without leaving the interface.
Third, the Security tab groups findings by root cause category (SQL injection, command
injection, hardcoded secret, and so on) rather than displaying them as a flat list
ordered by line number. Grouping by category lets a developer address all instances
of a single vulnerability class at once, which is typically how real remediation work
proceeds: fix every parameterised-query violation in a module rather than hunting them one line at a time.

trigger analysis of the changed files without posting to GitHub, review the findings

optionally post the results as inline review comments with a second confirmed action.

via \texttt{github.ts}, so no manual token entry is required after the initial
connection in Settings.

\section{Implementation Challenges}
\label{sec:challenges}

Several challenges that came up during development are worth documenting because they would
affect any similar ML-backed web application.

\subsection{Pydantic v2 compatibility}

The backend was originally prototyped against Pydantic v1 idioms, where model output
objects were accessed as dictionaries (\texttt{model.get("field")}). When dependencies
were pinned to Pydantic v2, this pattern silently returned \texttt{None} for every field
because Pydantic v2 output objects are not dicts. The failure showed up only at
runtime during the \texttt{/analyze} endpoint, not at model load time, which made it
hard to isolate. The fix required systematically replacing every dictionary-style
field access with attribute access (\texttt{model.field}) across all model output
handlers in \texttt{main.py}. The episode reinforced one specific lesson: validate
response schemas at endpoint boundaries; do not rely on duck typing to surface
type mismatches.

\subsection{Streaming and browser push}

Implementing real-time streaming via the \texttt{/analyze/stream} endpoint required
choosing a server push mechanism. WebSockets would have required a stateful session,
which cuts against FastAPI's stateless request model. Long-polling was ruled out
because it generates unnecessary HTTP overhead for incremental results.
Server-Sent Events fit: they run over standard HTTP/1.1 and require no additional
library on the server. The practical snag was that FastAPI's default
\texttt{JSONResponse} does not support streaming. The endpoint uses
\texttt{StreamingResponse} with \texttt{text/event-stream} MIME type and a generator
function that yields named events as each model completes. Because the endpoint
accepts a POST body (the code submission), the browser's native \texttt{EventSource}
API cannot be used directly—it only supports GET. The frontend instead opens a
\texttt{fetch} request and reads the response body through the Streams API
(\texttt{ReadableStreamDefaultReader}), parsing SSE frames manually and merging
partial results into the accumulating analysis state through React's reducer pattern.

\subsection{Cache eviction and memory bounds}

The in-memory hash-keyed analysis cache accumulates entries without bound unless
actively managed. Each cache entry stores the analysis result alongside a Unix
timestamp recorded at insertion time. When the entry count reaches the limit of 200,
the entry with the earliest timestamp is evicted before the new result is written.
An earlier version used \texttt{collections.OrderedDict} and popped from the front,
which evicts in insertion order, equivalent to timestamp order only when entries are
never accessed out of sequence. In practice, the admin \texttt{POST /cache/clear}
endpoint resets the entire cache at arbitrary points, leaving the insertion-order
assumption unreliable. The corrected implementation calls
\texttt{min(store, key=lambda k: store[k][``ts''])} to identify and remove the
genuinely oldest entry regardless of access pattern. The \texttt{POST /cache/clear}
button on the Admin page remains available for administrators who want to flush the
cache entirely.

\subsection{Input placeholder ambiguity}

An unexpected usability issue arose during testing of the PR Review page. The
original placeholder text for the URL input field was
\texttt{https://github.com/owner/repo/pull/123}, which is the standard format used in
documentation examples. In practice, placeholder text rendered in a muted colour is
easy to misread as actual content, particularly on bright screens. Several test sessions
showed users clicking the ``Analyze PR'' button with an empty input, expecting analysis
to start, and concluding the feature was broken when it returned a validation error.
The fix was to replace the URL-shaped placeholder with a plain instruction:
\textit{Paste your GitHub pull request URL here\ldots}. A natural-language prompt
cannot be mistaken for a valid URL. The lesson is specific to inputs whose placeholder
format is visually indistinct from real data: any placeholder that looks like a filled
field will occasionally be treated as one.

\subsection{Scenario 1: detecting vulnerabilities in an authentication module}

The first scenario illustrates the security detection pathway. Consider a short
Python authentication helper that constructs a database query by embedding a username
directly into an f-string and stores an API key as a module-level string literal. Both
patterns are extremely common in early-stage or tutorial-quality code, and both carry
serious real-world consequences if they reach production.

When this file is submitted, IntelliCode Review assigns an overall score of 37 and
raises a \textit{Critical} status badge. The security subsystem flags two findings: the
module-level API key (CWE-798: use of hard-coded credentials) at the line where the
string literal appears, and the f-string SQL construction (CWE-89: SQL injection) at
the query assembly line. The RF+CNN ensemble places the vulnerability probability at
0.95, well above the operating threshold of 0.050 derived from the precision-recall
curve during training.

The contrast between security and complexity scores is what makes this scenario useful. The function itself is short and straightforward: cyclomatic complexity of 4, cognitive complexity of 6, an overall complexity grade of A. In other
words, the code is easy to read but deeply insecure. This separation of signals is only
possible because the platform treats security and structural quality as independent
dimensions rather than collapsing them into a single score. A developer consulting only
a traditional complexity tool would see a clean bill of health; IntelliCode Review
surfaces the critical risk even for structurally simple code.

The Fix Guide tab renders two remediation blocks for this file: a parameterised query
template using Python's \texttt{sqlite3} placeholder syntax, and a pattern that reads the
credential from an environment variable via \texttt{os.environ.get}. The developer can
copy either snippet directly and adapt it, without needing to look up the correct API
for parameterised queries. Technical debt for this submission totals 240 minutes at
grade D, reflecting the two critical-severity findings, each of which contributes 120
minutes by the SQALE debt model.

\subsection{Scenario 2: structural debt in a secure but complex module}

The second scenario examines the opposite situation: a function that passes all security
checks but exhibits structural problems that will complicate future maintenance. The
function in question is a generic record-filtering utility that accepts a list of records
along with optional parameters for filtering, sorting, pagination offset, and result
limit. The logic is correct and the function is not vulnerable (the security model reports a vulnerability probability of 0.18 and raises zero findings), but the implementation compresses several independent concerns into a single body, producing a nested conditional structure that is hard to follow.

IntelliCode Review scores this submission 65 (grade C, \textit{Action Required}).
Cyclomatic complexity reaches 9 and cognitive complexity reaches 14, both driven by the
double-nested loop that handles multi-key filter matching. The platform generates three
refactoring suggestions: Extract Method for the inner filter comparison logic, Reduce
Nesting by flipping the conditional guard, and Reduce Parameters by grouping the
optional arguments into a dataclass or named tuple. Estimated remediation effort is
45 minutes at technical debt grade B.

The pattern classifier assigns a moderate-confidence \textit{feature envy} label,
recognising that the function's heavy use of record field accesses resembles a method
that belongs closer to a Record class than to a general utility module. This observation
is the kind of design-level feedback that static metrics alone cannot provide; the
trained classifier surfaces it because functions with similar structural profiles in the
training data were consistently labelled with the same smell.

This scenario is a useful counterpoint to the first. It demonstrates that the value of
a multi-dimensional analysis platform lies not in any single alarm being raised, but in
the full picture: a file can be secure and still carry meaningful structural debt that
will compound over time if left unaddressed.

\subsection{Scenario 3: validating a refactoring through the Compare page}

The third scenario follows the natural next step after Scenario 2: the developer acts on
the refactoring suggestions and wants to confirm that the changes actually improved the
code before opening a pull request. The refactored version extracts the inner filter
loop into a private helper function and groups the optional parameters into a
\texttt{QueryOptions} dataclass, separating pagination and sorting concerns from the
filter logic itself.

Both the original and refactored versions are submitted simultaneously via the Compare
page, which dispatches two parallel calls to \texttt{POST /analyze} and renders a
colour-coded delta table. The outcome confirms the improvements: complexity grade rises
from C to B, cognitive complexity drops from 14 to 6, and the refactoring suggestion
count falls from three to one. Only the Reduce Parameters suggestion remains, because the dataclass, while an improvement, still exposes multiple fields. Technical debt
improves from grade B to grade A. The overall score rises from 65 to 78.

What the Compare page provides that a manual code review cannot is a quantitative,
reproducible benchmark. Instead of relying on a reviewer's subjective impression that
the refactored version ``looks cleaner,'' the developer has concrete scores that can be
committed alongside the change and referenced in the pull request description. This
closes the feedback loop that motivated the platform's design: developers should be
able to measure the impact of their decisions, not just receive recommendations.

\section{Evaluation methodology}
\label{sec:eval-methodology}

Evaluating a system like IntelliCode Review presents a genuine methodological challenge.
The platform operates across four distinct machine learning tasks (regression, binary classification, multi-class classification, and sequential pattern recognition) as well as eight deterministic analysis modules and a real-time web interface. No single metric
or evaluation protocol captures this breadth. The strategy adopted here addresses the
three research questions from Section~\ref{sec:rq} at different levels of evidence: model
performance on held-out data (RQ2), end-to-end API correctness and latency (RQ1, RQ3),
and cross-project generalisation measured through Leave-One-Project-Out (LOPO) benchmarks.

\textbf{Scope of accuracy evaluation.} Precision and recall evaluation is performed
for the four ML-driven modules only: security detection, complexity prediction, bug
prediction, and pattern recognition. The eight deterministic modules (clone detection,
dead code, refactoring suggestions, technical debt, documentation, performance hotspots,
dependency coupling, and readability) are validated for pipeline correctness — they
produce outputs in the expected schema and the routing is correct — but not for
detection accuracy. Spot-checks for five of the eight deterministic modules (dead code, refactoring,
clone detection, documentation quality, and readability) are reported in
Section~\ref{ssec:multi-language-adapter}. Technical debt, performance hotspot, and
dependency coupling evaluators require project-level build artefacts not available in
snippet form; systematic evaluation for those three modules remains future work.
Claims about the system's analysis breadth (twelve dimensions) refer to the number of
dimensions covered, not to validated accuracy across all twelve.

\subsection{Correcting a label leakage defect}
\label{ssec:audit}

The results reported here differ from the original implementation in one respect that must
be documented before interpreting any numbers. A peer-review audit found a methodological
error in the original complexity model. The original complexity
model used the Maintainability Index (MI) as the regression target. This was a critical
error: MI is computed analytically from cyclomatic complexity, Halstead volume, and
lines of code, the same quantities that form the feature vector. In effect, the model
was being asked to learn a deterministic function from its own inputs, producing a
spurious $R^2$ of 1.000 that would never replicate on out-of-distribution code.

The corrected model uses \textit{cognitive complexity} as the target. Unlike MI, cognitive
complexity is computed by a separate AST traversal pass that counts structural nesting
penalties independently of the metric extraction pipeline. The feature vector is reduced
to fifteen dimensions by explicitly excluding cognitive complexity, which would otherwise
reintroduce leakage as a near-perfect predictor of itself. This correction brings the
in-distribution $R^2$ down to 0.712, a result that reflects genuine predictive signal
rather than tautological self-prediction.

A second leakage defect was independently identified in the pattern classifier. The
original 26-feature model was trained on cyclomatic complexity (CC), SLOC, Halstead
volume, Maintainability Index, and related metrics, the same quantities used by the
label assignment function to decide whether a function is \texttt{anti\_pattern}
($CC > 20$), \texttt{clean} ($\text{SLOC} \leq 40$), or \texttt{style\_violation}
(lines over 80). Spearman correlation between features and class indicators made the
circularity concrete: $|r| = 0.68$ between CC and \texttt{anti\_pattern},
$|r| = 0.71$ between SLOC and \texttt{clean}. A 5-fold stratified CV comparison
quantified the inflation: full 26-feature F1 $= 0.748$ vs.\ debiased 12-feature
F1 $= 0.665$ (Wilcoxon $W=0$, $p = 0.031$), a statistically significant
$+0.082$ inflation from circular supervision. The debiased model was retrained on
12 purely structural AST features (counts of functions, classes, nesting depth,
parameters, decorators, imports, and function body lines) and is the production
checkpoint. The honest CV F1 is $0.657 \pm 0.016$.

The deeper issue is that inflated metrics generated by leaking the target into the
feature space give practitioners false confidence in a deployed system. All results
from this point forward are from the corrected models.

\subsection{Statistical evidence and its limits}
\label{ssec:stat-limits}

Where sample sizes permit, this thesis reports formal confidence intervals and
hypothesis tests. For multi-seed in-distribution evaluations ($n = 5$ seeds), standard
deviation across seeds quantifies run-to-run variance. For the pattern leakage comparison,
a Wilcoxon signed-rank test over 5-fold CV scores yields a concrete $p$-value ($p = 0.031$).

The LOPO cross-project benchmarks operate under a stricter constraint: only four projects
are available for security and pattern tasks, and ten for complexity. At $n = 4$,
the Wilcoxon signed-rank test cannot achieve $p < 0.0625$ regardless of effect size,
so conventional $\alpha = 0.05$ significance is structurally unattainable. Rather than
reporting misleading $p$-values, this thesis uses Cohen's $d$ effect sizes and
\textit{consistency of direction across folds} as the primary evidence of robustness.
A medium effect size ($d \approx 0.5$) with consistent direction across all folds is
treated as meaningful evidence; a large effect with mixed direction is not. Readers
should interpret all LOPO comparisons under this framework rather than expecting
traditional significance thresholds.

\subsection{In-distribution evaluation and multi-seed variance}

The primary in-distribution evaluation uses held-out 20\% test splits drawn from each
task's real open-source dataset. For the complexity regressor, the principal metrics are
$R^2$ (explained variance), RMSE (scale-dependent error), and Spearman rank correlation
$\rho$ (ranking fidelity, which matters more than absolute accuracy for a tool that
assigns grade letters). For the binary security and bug classifiers, AUC-ROC is the
primary metric because it is threshold-independent and relatively robust to class
imbalance; it is supplemented by average precision (AP) to characterise performance
in the high-recall region. For the four-class pattern recogniser, macro-averaged F1 is
reported alongside Cohen's kappa, which corrects for agreement by chance and is more
appropriate than raw accuracy when class distributions are unequal.

To quantify the sensitivity of results to random initialisation, three of the four
trained models (security, complexity, and bug prediction) are evaluated across five independent seeds $\{42, 0, 7, 123, 999\}$, and results are reported as mean $\pm$
standard deviation. The pattern recognition model uses a fixed seed for its Random
Forest and relies on cross-validation variance rather than seed-resampling for
stability assessment. This matters because a result obtained from
a single seed may reflect a favourable or unfavourable random split rather than a
stable property of the model.

The four training datasets differ in size, source diversity, and label quality.

Three limitations deserve upfront acknowledgement rather than relegation to a threats
section. They are not defects to be defended but constraints to be reported.

\textbf{Domain coverage.} The original security, complexity, and pattern corpora drew
exclusively from Python web frameworks: Django, Flask, requests, pip, and SQLAlchemy.
Web-framework code has characteristic patterns (high decorator density, ORM-heavy database access, request-response handler structure) that are largely absent from
scientific computing, CLI tools, and infrastructure code. To address this, the datasets
were expanded using 33 repositories spanning five domains: web frameworks, data science
(NumPy, pandas, scikit-learn, matplotlib), testing infrastructure (pytest, Hypothesis),
CLI and developer tools (httpie, rich, pylint, flake8), and async/networking libraries
(aiohttp, Starlette, asyncpg, urllib3). The expanded complexity corpus reaches 3,015 samples; the pattern corpus reaches over 6,200 samples. Whether
this expansion is sufficient to remove domain bias is assessed in the LOPO results.

\textbf{Label quality.} Security labels derive from commit-message keyword matching
(``CVE'', ``security'', ``vulnerability''). This is a known-noisy procedure: commits that
fix security issues silently produce false negatives, and commits that mention
``security'' without introducing a fix produce false positives. The LAMV instrument
applied in Section~\ref{sec:novel-contribs} flags leaky features using oracle AUC
comparison but cannot correct label noise in the underlying commits. Bug labels have
the same problem: the SZZ keyword heuristic identifies commits whose messages mention
``fix'', ``bug'', or ``error'', which captures real bugs but also routine maintenance
commits. The failure taxonomy in Section~\ref{ssec:failure-analysis} identifies 19
label-noise cases in the security LOPO folds as the dominant failure mode.

\textbf{Scale.} Even after expansion, 4,286 to 8,668 samples per task is on the
smaller end of the defect prediction literature. The LOPO protocol partially compensates:
ten independent train-test splits are more stringent than a single held-out split of a
larger homogeneous dataset. But a model trained on 4,000 samples cannot learn the full
distributional complexity of professional production code. The security LOPO AUC of 0.494
(essentially chance) is an honest reflection of this.

\subsection{Cross-project generalisation via Leave-One-Project-Out}

In-distribution evaluation, even with a held-out split, cannot guarantee that a model
will perform on code written in a different project or codebase. Functions from the same
repository share naming conventions, design patterns, and domain-specific idioms that
leak implicitly into both training and test splits. To assess genuine cross-project
transfer, this evaluation uses a Leave-One-Project-Out (LOPO) protocol for complexity:
each of ten OSS repositories is held out in turn, the model is retrained on functions
from the remaining nine, and performance is measured on the held-out project. This
eliminates within-project bias by construction.

Spearman rank correlation is preferred over RMSE as the primary LOPO metric because
RMSE is sensitive to the absolute range of cognitive complexity values in the held-out
project, which varies considerably across repositories. A model that correctly ranks
functions from simple-to-complex within a project is useful for priority-ordering
refactoring work, even if the absolute scores differ from a reference implementation.

\subsection{Temporal split for bug prediction}

Bug prediction is particularly susceptible to a form of temporal leakage that is easy
to overlook. When commit histories are split randomly, commits that post-date a bug fix
can appear in the training set with a label indicating the file is ``buggy,'' while the
commit that introduced the bug may end up in the test set. This creates an artificial
advantage for any model that happens to learn the temporal structure of the repository.

To detect and quantify this effect, the bug predictor is evaluated on both a standard
random split and a strict temporal split: the 80\% of commits with the earliest
timestamps are used for training, and the 20\% most recent commits are held out for
testing. The difference in AUC between the two conditions provides a direct estimate of
how much performance inflation the random split produces.

\subsection{Calibration against SonarQube}
\label{ssec:sonarqube-cal}

SonarQube is the de facto industry standard for cognitive complexity measurement. Aligning IntelliCode Review's complexity metric with SonarQube's definition determines whether scores are interpretable in the context developers already use.

The calibration was performed on fifteen hand-crafted benchmark functions designed to
cover the full range of structural patterns penalised by cognitive complexity: simple
conditionals, nested conditional chains, for-else constructs, while loops, mutual
recursion, and nested inner function definitions. Reference values were computed
manually following the SonarQube specification, and the implementation's outputs were
compared against these references using Pearson correlation, MAE, and bias.

\subsection{Conformal prediction and uncertainty quantification}

Point predictions of cognitive complexity may be accurate on average but unreliable
for individual functions. Conformal prediction provides a principled way to convert
these point predictions into intervals with guaranteed finite-sample coverage, under
the assumption of exchangeability between calibration and test data.

Residual-based conformal intervals are calibrated on a held-out calibration split
(separate from both training and test data). Coverage is verified empirically at the
80\%, 90\%, and 95\% nominal levels. An interval is considered valid if its empirical
coverage meets or exceeds the corresponding nominal level.

\subsection{End-to-end API correctness}
\label{ssec:api-testing}

Beyond model accuracy, a deployed system must behave correctly as an integrated unit.
A suite of 24 structured test cases was constructed to exercise the full analysis
pipeline through its HTTP interface. The cases span four categories: eight
known-vulnerable code snippets drawn from CWE examples (one per vulnerability type
supported by the security model), four known-clean Python functions, six boundary cases
for the complexity grader, and six dead code detection cases. Each test case
asserts both the HTTP status code and the semantic content of the response, for example that a SQL injection example produces at least one finding with severity ``critical''
and a CWE-89 identifier.

\subsection{Latency benchmarking}
\label{ssec:latency}

The platform's practical utility depends on whether it is fast enough to fit into a
developer's editing workflow. Analysis results that arrive after ten or fifteen seconds
are unlikely to be consulted as frequently as results that appear within two seconds.
Latency for \texttt{POST /analyze} was measured across five input sizes (10, 50, 100, 500, and 1000 source lines of code) using 20 repeated requests per size on a
development machine (Windows 11, Intel Core i7, 16\,GB RAM, CPU-only PyTorch without
GPU acceleration). Measurements are reported as mean, 95th-percentile, and maximum.

The complexity model achieves a 5-fold CV mean of $R^2 = 0.676 \pm 0.063$ (RMSE = $28.2 \pm 2.7$, Spearman

Cognitive complexity is influenced by code structure in ways that volume and cyclomatic
metrics capture only partially, so a gap between the model's output and the true score
is expected rather than surprising~\cite{mccabe1976}. The Spearman correlation of
$0.819 \pm 0.027$ is the more practically relevant figure: the model's ranking of functions from simple to complex is stable and reliable, which
is what matters when a developer wants to know which file deserves attention first.

In-distribution, the security model works well. The five-seed AUC of $0.963 \pm 0.012$ is achieved after expanding
the training corpus from 1,286 to 4,286 samples by mining additional clean production
code from the same OSS repositories used for LOPO evaluation. The larger, better-balanced
dataset substantially reduces variance and improves in-distribution discriminability.
The RF+CNN ensemble combines the feature-based stability of the Random Forest with the
token-sequence sensitivity of the 1D convolutional network; the ensemble consistently
outperforms either component alone. Average precision of $0.653 \pm 0.078$ is lower
than AUC because the positive class remains a minority in the expanded dataset, and AP
penalises false positives more heavily at high-recall operating points. The operating
threshold was updated to 0.050 on the expanded training precision-recall curve.

The bug predictor AUC of $0.676 \pm 0.038$ is broadly consistent with the JIT defect
prediction literature~\cite{lessmann2008, kamei2013}, where SZZ-style commit labels are
known to be noisy. A commit message containing the word ``fix'' does not always
correspond to a genuine defect repair, and some genuine fixes use different vocabulary.
This label noise sets a practical ceiling on what any metric-based model can achieve
regardless of architecture.

The debiased pattern classifier achieves CV F1 $= 0.657 \pm 0.016$ and test F1 $= 0.683$
(AUC $= 0.846$, accuracy $= 66.5\%$). These are lower than the original 26-feature model's
numbers (CV F1 $= 0.735$, test F1 $= 0.804$), and they should be: the
$+0.082$ gap is the leakage inflation measured by the Wilcoxon test ($p = 0.031$),
not genuine predictive improvement. The debiased figures reflect what the model actually
contributes beyond memorising the label assignment rules. A construct-validity caveat
remains: even the 12 structural AST features are not completely orthogonal to the
label oracle, since features like \texttt{n\_functions} and \texttt{max\_function\_body\_lines}
may correlate weakly with the oracle's thresholds. The only definitive fix is a
human-annotated smell dataset independent of any automated oracle. The debiased model
is the production checkpoint; its predictions should be treated as calibrated soft
signals rather than authoritative classifications.

The model fails on \textit{tiangolo/fastapi} ($\rho = 0.255$). FastAPI relies heavily
on Pydantic validators, Python decorators, and dependency injection, all of which produce
ASTs structurally different from the request-handler and utility code that dominates the
training set. At the feature level, the failure is specific: decorator-chained route

complex, so the features the model relies on most (cyclomatic complexity, Halstead volume, and nesting depth) systematically underestimate function complexity in this style. The model's top features have no way to represent that a function's apparent
simplicity is an artifact of decorator indirection rather than genuine simplicity.

This is not a marginal case. FastAPI is one of the most widely used Python web
frameworks, and its annotation-driven style is increasingly common across the ecosystem.
The training corpus (Django, Flask, requests, pip) is older and more imperative; any
codebase that leans on decorators, async patterns, or Pydantic-style validation will
push the model past its training distribution. Developers working on such projects
should treat complexity scores as rough estimates and rely instead on the deterministic modules (refactoring suggestions and dead code detection) that do not depend on the learned model.

The mean RMSE of $45.6 \pm 26.2$ looks alarming in isolation but is better understood
in context. The wide standard deviation is driven by \textit{pydantic/pydantic}, where
deeply nested validators push absolute complexity values far above the training
distribution, yielding RMSE = 107.6. Yet even there, the Spearman correlation remains
0.832, meaning rankings are preserved even when absolute values drift. This reinforces
the choice to use Spearman correlation as the primary LOPO metric.

\textbf{Feature importance and the psf/requests case.}
XGBoost's built-in gain-based feature importance for the complexity model ranks
\texttt{n\_complex\_functions} (functions with CC $> 10$) as the dominant feature,
accounting for approximately 62\% of split gain. \texttt{n\_long\_functions} and
\texttt{max\_func\_cc} contribute a further 15\%.
Cognitive complexity, excluded as the prediction target, would otherwise dominate.
This ordering explains both the model's success and its failure mode. Repositories
with complexity driven by control-flow branching (conditionals, loops, exception
handling) map cleanly onto the feature set. Repositories where complexity is
architectural rather than control-flow-visible (encapsulated in decorators, dependency injection chains, or deep class hierarchies) do not.

\textit{psf/requests} is an instructive intermediate case ($\rho = 0.727$, the weakest
result outside fastapi). The requests library uses a clean imperative style at the
function level, but its API surface involves a large number of short, shallow functions
that perform dispatch or option handling, functions the model assigns low complexity scores to correctly, but whose interactions create emergent difficulty that only
appears at the call-graph level. The ranking is directionally correct within the
repository but noisier than in codebases where function complexity is more
self-contained.

For security, the LOPO picture is less positive. Mean AUC of $0.494 \pm 0.189$ is
essentially at chance level, indicating that the security model's vulnerability
detection relies on patterns that are specific to the CVE-annotated training projects
and do not transfer reliably when the distribution of vulnerability types shifts. The
bug predictor performs marginally better across three held-out repositories (mean AUC
$0.567 \pm 0.028$), but remains well below the in-distribution figure of 0.676. These
cross-project gaps show that domain adaptation is a genuine unsolved problem for both tasks.

\subsection{Temporal split for bug prediction}
\label{ssec:temporal}

The gap between the random-split AUC of 0.618 and the temporal-split AUC of 0.460
(a delta of $-0.158$) is both statistically meaningful and practically important. It
confirms that the standard random-split evaluation overstates true predictive ability
by approximately 24\% in relative terms. This inflation arises because commits that
post-date a bug fix can appear in the training set, inadvertently teaching the model
temporal patterns that are not available at true inference time, namely when a developer wants to know whether the code they are writing \textit{today} is at risk of future bugs.

The temporal AUC of 0.460 is technically below chance (0.5), a result that initially
appears paradoxical but is consistent with published findings on JIT defect
prediction~\cite{kamei2013}: when evaluated strictly on future commits, models that
learn the temporal signature of recent bug-fix activity actually develop a slight
\textit{negative} correlation with future bug probability, because recently fixed files
tend to be cleaner than the population average. This phenomenon, where feature--label correlations learned during training invert under the deployment distribution and cause systematically wrong predictions, is termed \textit{anti-learning}
in the distribution shift literature. It occurs here because the static metrics that
correlate with historical bug density (high complexity, many contributors) are
\textit{reduced} after a bug fix, making recently-fixed high-complexity files
appear safer than they are. This is the anti-learning effect in concrete form: the signal is real but fragile, and
an honest temporal evaluation strips away much of what appears to be signal in a random
split. The same pattern appears in published JIT defect prediction work that uses a
temporal boundary~\cite{kamei2013, d2010}; the random-split AUC of 0.676 is consistent
with the field for static-metric models.

To address the single-boundary limitation, the evaluation was extended with a
\textbf{5-fold walk-forward temporal cross-validation}. The commit timeline is divided
into five equal slices sorted chronologically. For fold $k$ ($k = 1, \ldots, 4$),
the model is trained on slices $0$ through $k-1$ and tested on slice $k$ (the next
time window). This expanding-window protocol removes the single-cut sensitivity and
tests the model at four distinct points in the timeline.

\centering

\begin{tabular}{@{}lll@{}}
\toprule
\textbf{Fold} & \textbf{Train slices} & \textbf{Test AUC} \\
\midrule

The mean walk-forward AUC is $0.544 \pm 0.105$, barely above chance, consistent with
the single-boundary result. The high fold variance ($\pm 0.105$) is itself informative:
the model's temporal performance is sensitive to \textit{which} window is used as the
test set, confirming that no single cut point is representative. Folds 1 and 4 fall
below chance; folds 2 and 3 are modestly above. This pattern is consistent with
anti-learning: the model occasionally aligns with the forward signal but cannot do so
reliably. The direction (well below the in-distribution AUC of 0.676) is stable across all four folds and matches the published literature on static-metric JIT models.

Five slices yield four training windows, not five, because fold 1 requires slice 0 as
its sole training set and cannot be tested on itself. Adding more cut points would
raise the fold count but would not address the high variance: the earliest folds
already train on fewer than 1,500 commits, a sample size at which XGBoost is
learning noise rather than signal. The variance is a symptom of the labelling noise
and data scarcity discussed above, not of the evaluation design. A larger, more
consistently labelled dataset is the prerequisite for a stable walk-forward estimate,
not a finer time grid.

Practically, the bug predictor should not be deployed as a standalone forecast of future
bug risk. What it measures is structural complexity: high scores flag files that are
hard to read and modify, which is useful context even if it does not predict which files
will attract bugs next quarter. As one signal among twelve, with this limitation
disclosed to the user, it contributes something without overstating what that something
is.

The calibration results are reassuring. Across fifteen benchmark functions covering

correlation indicates that the two implementations agree on which functions are complex

The positive bias of $+1.07$ points traces to a single implementation difference:
the SonarQube specification applies a nesting increment only when a function
definition appears \textit{inside} another function that is itself non-trivial, whereas
the IntelliCode Review implementation increments for any nested function definition.
The effect is systematic rather than random. Three considerations justify not applying
a post-hoc correction in the current system. First, the bias is constant across all
inputs: because every function's score shifts by the same amount, relative rankings are
unaffected. The Spearman correlation of 0.957 confirms this directly: IntelliCode Review and SonarQube agree on which functions are most complex and which are simplest.
Second, the grade boundaries (A through F) were calibrated against the uncorrected
scores; applying $-1.07$ would require shifting all thresholds simultaneously to
preserve grade assignments, creating a risk of regression for existing users. Third,
the system is not positioned as a drop-in SonarQube replacement: its grade letters are
internal quality signals relative to one another, not claims of numerical equivalence
with any external tool. If cross-tool numerical agreement becomes a hard requirement, the
correction can be applied in a single line of the complexity scorer without affecting
any other component.

The practical upshot: a developer moving between SonarQube and IntelliCode Review will
see scores that differ by roughly one point on average, but the relative ordering of
functions will be the same. For a tool whose purpose is to tell you \textit{which files
to look at first}, that is what matters.

prediction should not be trusted, rather than as a precise bound on the true complexity

The result establishes the proximate cause of the LOPO collapse: removing project-style
features alone does \textit{not} bring LOPO AUC above 0.55. Zeroing out call and import
counts reduces the mean from 0.530 to exactly 0.500 (chance), with a standard deviation
of zero across all 31 folds, indicating the model outputs constant probabilities — it
cannot discriminate at all without those features. Size proxies (bytes, lines) have no
effect on their own.

The deeper cause — \textit{why} call and import counts are non-transferable — is that
these features reflect project-level coding conventions rather than local vulnerability
semantics. The number of function calls in a file and the number of imports are
determined primarily by the codebase's module structure: a Flask application with
one handler per file has a different typical \texttt{n\_calls} distribution than a
SQLAlchemy ORM module with hundreds of cross-function calls, regardless of whether either
contains a vulnerability. The RF model learns these style fingerprints and uses them as
implicit project identifiers. On held-out projects it has never seen, those fingerprints
differ from the training distribution, and the features carry no vulnerability signal.

To test whether style-feature shift magnitude explains the per-project variance in LOPO
AUC, a dedicated analysis was run (\texttt{backend/evaluation/security\_lopo\_shift\_analysis.py})
computing Jensen-Shannon divergence between each project's \texttt{n\_calls},
\texttt{n\_imports}, and SLOC distributions and the training pool, then correlating
those shift scores with the per-project LOPO AUC across the 21 matched projects from
the 31-project benchmark. The result is $r = +0.13$ ($p = 0.58$, $n = 21$) — not
significant in either direction. The Shift-Aware Evaluation Framework (SAEF) result
of $r = -0.40$ reported in Section~\ref{sec:novel-contributions} was computed on
only $n = 3$ projects and is insufficient to draw conclusions.

The $n = 21$ finding strengthens the diagnosis rather than weakening it: the
style-confounding is \textit{uniform} across projects, not varying by shift magnitude.
All projects are affected roughly equally because the model's only discriminative signal
under LOPO (\texttt{n\_calls} and \texttt{n\_imports}) is a style fingerprint that
carries no cross-project vulnerability information in any project. Individual variation
in LOPO AUC is driven by label distribution differences across projects — some
repositories have more consistent labelling between their vulnerable and clean functions
— rather than by how far a project's feature distribution lies from the training centroid.
The conclusion remains the same: call and import counts are style proxies, not
vulnerability indicators, and interprocedural taint flow or function-call graph features
are required to move the LOPO ceiling substantially above chance.

Fourth, for the bug predictor, only static features ($-0.036$) and JIT size-change
features ($-0.028$) contribute meaningfully. The remaining four JIT feature groups
(diffusion, purpose, author history, developer experience) have zero marginal effect,
suggesting that the bug predictor's weak cross-project performance stems partly from
relying on features that are informative in-distribution but non-transferable.

files at the p95 tail. The relationship between input size and latency is approximately
linear, which reflects the AST-walking modules dominating the overall budget: clone
detection and dead code analysis both traverse the syntax tree at $O(n)$ in the number
of AST nodes. The ML inference itself contributes less than 30 milliseconds across all
input sizes, because the feature extraction step produces a fixed-size 15-dimensional
vector regardless of file length.

The \texttt{/analyze/quick} endpoint, which skips the eight deterministic modules and
runs only security pattern matching, complexity scoring, and readability analysis,
returns results in 200--400\,ms for typical 100-line files. This makes it viable for
integration into a CI step with a 500\,ms budget, and the SSE streaming endpoint
delivers the first results within 200\,ms of the request being received, giving a
perceptibly fast response even when the full analysis takes longer.

structural content. XGBoost also provides calibrated conformal prediction intervals

the absence of better size-independent features; the rank-correlation underperformance
relative to the trivial baseline is a real and acknowledged limitation.

Radon and the LOC baseline cannot learn from examples or adapt to project-specific

IntelliCode Review trades some rank-correlation performance for calibrated point
predictions, conformal intervals, and multi-feature generalisation.

\section{Evaluation protocol components}
\label{sec:novel-contributions}

A standard held-out test split and an AUC score cannot surface the failure modes documented in this thesis. Label leakage, cross-project generalisation, calibration degradation, and actionability each require their own instrument; none of the four appears in a standard evaluation toolkit. Ten such instruments were developed and applied in this work. All ten are implemented in \texttt{backend/evaluation/} and run via \texttt{backend/run\_novel\_contributions.py}; results persist to \texttt{evaluation/results/} as JSON files.

\subsection{Generalisation diagnostics}

The first group of contributions addresses a problem that in-distribution metrics
cannot see: whether a model's learned features transfer across projects or degrade
once the deployment code differs from the training distribution.

\textbf{Project-Invariant Feature Filtering (PIFF)} identifies features whose
distributions are stable across projects and restricts the model to those features
at evaluation time. Features with high cross-project coefficient of variation carry
project-level style information rather than transferable code quality signal.
Prior work on cross-project defect prediction (Menzies et al.~\cite{menzies2011},
Nam et al.~\cite{nam2013}) addresses distribution shift through domain adaptation or transfer
learning; PIFF is a simpler upstream filter that improves generalisability without
retraining, making it applicable as a pre-evaluation check before any adaptation
strategy is chosen.
On the four-project LOPO benchmark, restricting the security model to PIFF-selected
features raises mean AUC from 0.578 to 0.620, a gain of $+0.042$ (Cohen's $d = 0.558$,
medium effect). Statistical significance is unattainable at $n = 4$ projects, but the
direction is consistent across all folds.

\textbf{Shift-Aware Evaluation Framework (SAEF)} quantifies how much the feature
distribution shifts between training and each held-out project, then measures whether
that shift predicts model failure. On the four-project initial LOPO benchmark, the
Pearson correlation is $r = -0.40$ ($p = 0.74$, $n = 3$), which is directionally
consistent with the hypothesis but statistically uninformative at that sample size.
A follow-up analysis on the 21-project security benchmark
(\texttt{security\_lopo\_shift\_analysis.py}) finds $r = +0.13$ ($p = 0.58$, $n = 21$):
style-feature shift magnitude does not significantly predict per-project LOPO AUC.
This is itself informative: the style-confounding is uniform across projects rather than
varying by shift magnitude, which means pre-screening projects by shift score would not
reliably identify which ones to trust. The value of SAEF is therefore as a diagnostic
instrument — it identifies which features drive the shift — rather than as a deployment
gating criterion at the current dataset scale. Existing data drift tools (e.g.,
Evidently, Alibi-Detect) measure shift after deployment on live traffic; SAEF is
designed as a pre-deployment audit, and the negative shift-AUC correlation finding is
a concrete output of running it.

\textbf{Cross-Project Calibration Monitoring (CACM)} tracks how well the model's
confidence estimates hold up outside the training distribution. In-distribution
Expected Calibration Error (ECE) for the security classifier is 0.0875, meaning the model's confidence scores roughly match its actual accuracy within the training domain.
Under LOPO evaluation, ECE rises to 0.2645, a threefold degradation. This means
that a security finding the model reports at 90\% confidence outside its training
distribution is substantially less reliable than that figure suggests. CACM makes
this degradation explicit rather than leaving it implicit in the AUC drop.

\textbf{Simplicity Baseline Comparison} tests whether a hand-engineered single
metric outperforms the trained model under cross-project evaluation. Under LOPO,
Halstead volume alone achieves AUC $= 0.586$ on the security task, marginally
exceeding the XGBoost model's LOPO AUC of 0.573. This is a deliberately contrarian
result: it documents a case where a 19th-century software metric beats a trained
ensemble when the training distribution does not match the test distribution.
The implication is not that ML is useless but that cross-project evaluation is
necessary to detect cases where it provides no advantage over simpler alternatives.

\subsection{Temporal robustness}

\textbf{Time-Decayed Code Prediction (TDCP)} applies exponential sample weighting
$w_i = \exp(-\lambda \cdot \Delta\text{years})$ during training to down-weight older
commits, reducing dependence on historical patterns that may not hold in the deployment
distribution. At $\lambda = 2.0$, temporal-split AUC improves from 0.430 to 0.440
($+0.010$). The gain is modest because label noise imposes a ceiling that weighting
alone cannot overcome, but the direction is positive at every tested $\lambda \in
\{0.5, 1.0, 2.0\}$. The 5-fold walk-forward cross-validation described in
Section~\ref{ssec:temporal} is a companion instrument that removes sensitivity to any
single temporal cut point. Both are in \texttt{tdcp.py}.
Instance weighting for temporal robustness appears in the JIT defect prediction
literature~\cite{kamei2013}, but prior implementations weight by recency within a
single project. TDCP applies the same idea cross-project, where the weighting is
calibrated against the observed temporal-split AUC degradation rather than an
arbitrary decay constant, and is evaluated via the walk-forward protocol rather than
a single temporal boundary.

\subsection{Leakage detection}

\textbf{Leakage-Aware Model Validation (LAMV)} is an automated pipeline that screens
feature-label correlations before training to detect target leakage. It flags any
feature with Spearman $|r| > 0.70$ against the label, then verifies by training an
oracle model using only that feature and checking whether it achieves near-perfect
AUC. Applied to the original bug predictor, LAMV correctly identifies \texttt{is\_fix}, a binary flag derived from commit message keywords, as leaky:
$|r| = 0.89$ against the label, oracle AUC $= 0.97$. Removing \texttt{is\_fix}
drops the random-split AUC from 0.93 to 0.676, which is the figure reported in
this thesis. LAMV is a pre-training check, not a post-hoc correction; it is designed
to catch these defects before a model is trained and reported.
Existing leakage detection work (e.g., Kaufman et al.~2012 on data leakage in ML competitions)
focuses on train/test contamination; LAMV targets a different and less-documented
failure mode specific to code analysis: features derived from the same heuristic
used to generate labels, which does not violate train/test boundaries but
corrupts the feature-label relationship throughout the dataset.

\textbf{Pattern Leakage Fix (PLF)} extends the leakage audit to the label assignment
function rather than the input features. For the pattern classifier, the original
oracle used function length and cyclomatic complexity to assign labels
(\texttt{anti\_pattern} if $CC > 20$, \texttt{clean} if $\text{SLOC} \leq 40$), then
trained a model on features that included those same quantities. PLF detected the
circularity through Spearman correlations between features and class indicators
($|r| = 0.68$ for CC versus \texttt{anti\_pattern}, $|r| = 0.71$ for SLOC versus
\texttt{clean}), quantified the inflation as $+0.082$ F1 via 5-fold stratified CV
comparison (Wilcoxon $W=0$, $p=0.031$), and produced the 12-feature debiased model
that is the production checkpoint.

\subsection{Actionability and failure analysis}

\textbf{Actionability Analysis (ACT)} measures whether security findings are useful
in practice, not just whether they are statistically discriminating. Two metrics are
reported. Proportion of Bugs in the top-20 findings (PofB@20) is 0.283: of the 20
highest-confidence security findings the model produces, 28.3\% correspond to genuine
vulnerabilities. False Discovery Rate at the 0.5 threshold (FDR@0.5) is 0.696: nearly
70\% of findings above the mid-range confidence threshold are false positives. These
numbers are not evidence of a broken tool; they reflect the difficulty of the task and the class imbalance in the dataset. They are the honest figures a practitioner
needs to calibrate how much review time to spend on each alert.

\textbf{Failure Taxonomy (FTAX)} applies a structured classification to every LOPO
failure case, assigning each to one of four root-cause categories: label noise,
distribution shift, feature scope mismatch, or oracle circularity. Across the LOPO
evaluation, 19 cases are classified as \textsc{label\_noise} (the ground-truth label
is unreliable for that sample) and 13 are classified as severe, meaning the model's
prediction is confidently wrong rather than merely uncertain. Having a named taxonomy
of failure modes rather than a single aggregate AUC figure gives practitioners a basis
for deciding which failures are correctable (label noise can be reduced with better
annotation) and which are structural (feature scope mismatch requires a different
model architecture).

\textbf{Statistical Significance Tests (SIG)} applies Wilcoxon signed-rank tests and
reports Cohen's $d$ effect sizes for all pairwise model comparisons. At $n = 4$
LOPO folds, the minimum achievable $p$-value is 0.0625, so significance at the
conventional $\alpha = 0.05$ threshold is unattainable regardless of effect size.
SIG makes this power limitation explicit in every comparison table rather than
suppressing it. The pattern leakage comparison ($p = 0.031$, $n = 5$ cross-validation
folds) is the one case where sufficient observations are available for a statistically
significant result; all others are reported with effect size and direction only.

\section{Discussion}
\label{sec:discussion}

\subsection{What the results collectively tell us}

The evaluation results follow a pattern that is consistent across tasks. The models do
well where the signal is structural and stable: nesting depth and cyclomatic branching are language-level properties, so complexity ranking transfers across most projects.
Security detection works within the vulnerability types the model was trained on and
fails when the held-out project has a different profile. Bug prediction degrades under
temporal evaluation because static features at commit time do not carry enough
information to distinguish future-buggy from future-clean code.

This pattern represents a genuine limitation, not simply a calibration problem to be
engineered away. The literature on software defect prediction has documented similar
boundaries for two decades~\cite{lessmann2008, kamei2013, d2010}, and the LOPO and
temporal results here are consistent with that body of work. The contribution of
IntelliCode Review is not to claim those limits have been overcome, but to measure
them honestly: by reporting LOPO AUC alongside in-distribution AUC, and by including
a temporal split that reveals the anti-learning effect, the evaluation gives
practitioners a more complete picture of what the models actually provide in deployment
than benchmark numbers alone would suggest.

\subsection{Why ML-based code analysis fails at the boundary}
\label{ssec:why-ml-fails}

The cross-project and temporal failures documented in Section~\ref{sec:model-results}
are not incidental. They point to structural reasons why ML models struggle with code
quality tasks in ways that do not apply to, say, image classification.

\textbf{Dataset bias.} All four models are trained on popular, well-maintained Python web frameworks (Flask, Django, requests, pip) written by experienced engineers who share
naming conventions, API idioms, and review culture. Code written by a solo contributor, in
a scientific computing context, or in a legacy enterprise codebase will not share these
surface patterns. That is not a data quantity problem; it is a distribution coverage problem
that more samples from the same sources cannot fix.

\textbf{Feature scope.} Every model here operates on single-function features: cyclomatic
complexity, line counts, Halstead volume, token sequences within one function body. Real
vulnerabilities are often interprocedural. A SQL injection requires untrusted input to
travel through several call boundaries before reaching a database sink. Single-function
features are blind to that flow. The model assigns a low vulnerability probability to a
function that safely formats a query string, unaware that the argument arrives
unsanitised from a route handler three files away. That is a mismatch between the
granularity of the analysis and the granularity of the problem.

\textbf{Label noise.} SZZ-style labels (marking a commit as buggy if a future commit message contains ``fix'' or ``bug'') introduce noise that caps performance regardless of model capacity. Many ``fix'' commits repair documentation, configuration, or test
infrastructure. Cognitive complexity as a regression target is better-grounded than the
Maintainability Index, but it still measures structural nesting penalties, not the
genuine difficulty a developer will experience reading the code. Every result in this
thesis is constrained by its weakest link, which is usually label quality.

\textbf{Temporal shift.} Code quality patterns evolve. A model trained on 2018--2022
commits will encounter different idioms and design conventions in 2025 code. The
temporal anti-learning result in the bug predictor illustrates this: a model that learned
that high-metric files are buggy becomes slightly anti-predictive once most of those
files have been cleaned up. Deployment does not pause for retraining.

None of these failure modes are unique to this platform. They appear across the software
engineering ML literature and are well documented in the defect prediction
community~\cite{lessmann2008, kamei2013}. Naming them here gives practitioners a
framework for deciding which modules to trust, under what conditions, and when to fall
back on the deterministic analysers that do not share these limitations.

Two further failure patterns were identified through targeted diagnostic examples, not
through a sampled evaluation; no error rates are reported because the examples were
constructed with knowledge of the model's feature set and are not representative of
production inputs. For the complexity model: Halstead volume and cognitive complexity
correlate on average, but the correlation breaks when structural complexity diverges
from token density. Deep nesting with trivial bodies has low Halstead volume but high
true cognitive complexity — the model underestimates. Expression-heavy functional code
(map/filter chains) has high Halstead volume but no branching penalty — the model
overestimates (worst observed: predicted CC = 13.8, true CC = 44). For the security
regex rules: false positives occur on clean code that contains vulnerability-pattern
substrings in comments or string literals — a known limitation of pattern-matching
consistent with the false positive rates of comparable tools in
Section~\ref{sec:market-comparison}.

\subsection{Partial mitigations for diagnosed failures}
\label{ssec:mitigations}

Diagnosing failure without attempting remediation is incomplete. Two targeted interventions
were implemented and evaluated against the failure modes identified above.

\textbf{Project-Invariant Feature Filtering (PIFF)} addresses the feature instability
problem. Features whose means vary widely across projects (high coefficient of variation
in per-project distributions) carry little cross-project signal; they reflect project-level style rather than underlying code quality. PIFF identifies the subset of features with
stable cross-project distributions and restricts the model to those features at test time.
On the four-project LOPO benchmark, PIFF features raise mean AUC from 0.578 to 0.620
($+0.042$, approximately 4 percentage points). Cohen's $d = 0.558$ indicates a medium
effect size. Statistical significance is unattainable with $n = 4$ projects (minimum
Wilcoxon $p = 0.0625$), but the direction is consistent: stable features generalise better.

\textbf{Time-Decayed Code Prediction (TDCP)} addresses temporal drift. Exponential
sample weighting $w_i = \exp(-\lambda \cdot \Delta\text{years})$ down-weights older
commits during training, reducing the model's dependence on historical patterns that may
no longer hold in the deployment distribution. At $\lambda = 2.0$, temporal AUC improves
from 0.430 to 0.440 ($+0.010$). The gain is small because the labelling noise (SZZ
keywords) imposes a ceiling: no weighting scheme can recover signal that was never in
the labels. The direction is positive at every tested $\lambda \in \{0.5, 1.0, 2.0\}$.

Together, PIFF and TDCP demonstrate that a meaningful fraction of the performance gap
between random-split and cross-project or temporal evaluation is attributable to feature
instability and temporal drift, not only to label noise. The remaining gap after both
interventions is not easily closed without higher-quality labels or more diverse training
repositories. The implementation is available in \texttt{backend/evaluation/piff.py} and
\texttt{backend/evaluation/tdcp.py}.

\subsection{Strengths and design goal fulfilment}

The evaluated system meets all five design goals stated in Section~\ref{sec:goals}.
It covers twelve independent analysis dimensions from a single interface, more than any
open-source tool available without dedicated infrastructure. Four of those dimensions use
trained ML models targeting code quality properties that rule-based tools cannot quantify:
complexity ranking, vulnerability probability, bug risk, and structural smell
classification. Full 1000-line analysis completes in under two seconds; the quick-path
endpoint stays below 500\,ms for CI integration. localStorage review history and
configuration survive page reloads without a backend database. The Fix Guide converts
each security finding into copy-ready remediation code the developer can apply without
leaving the interface.

The SonarQube calibration result has direct practical value.
Pearson $r = 0.980$ against an industry-standard reference implementation means that
developers already familiar with SonarQube's cognitive complexity scores will
find IntelliCode Review's numbers interpretable in the same frame of reference.

One boundary on these strengths must be acknowledged: the accuracy evaluation covers
only four of the twelve analysis dimensions. The eight deterministic modules (clone detection, refactoring suggestion, dead code identification, technical debt estimation, documentation quality, performance analysis, dependency coupling, and readability) are primarily validated through the 24 end-to-end API correctness tests, which verify correct
pipeline routing and output schema. To provide systematic accuracy estimates beyond
pipeline correctness, automated spot-checks were implemented for five of the eight
deterministic modules. Table~\ref{tab:det-spotchecks} summarises the results.

\subsection{Observational use during development}
\label{ssec:informal-validation}

Beyond the quantitative model evaluation, IntelliCode Review was applied repeatedly
to its own codebase during development. What follows are structured observations from
that use, not controlled experiment results.

The three demonstration scenarios in Chapter~\ref{ch:demo} are not contrived: the
authentication module in Scenario 1 (SQL injection via f-string, hardcoded credential)
is drawn from a real commit in a public open-source repository where both patterns
appeared together in a helper written for a tutorial. The tool flagged both findings
with the correct CWE identifiers and surfaced them at the correct line numbers. The Fix
Guide remediation blocks for that scenario were copy-pasted directly into the corrected
version of the file with no modification.

During development of the backend itself, IntelliCode Review was applied repeatedly to
\texttt{backend/main.py}, \texttt{backend/features/ast\_extractor.py}, and the training
pipeline. It caught two dead code cases (an unused import and an unreachable branch in
the complexity model's grade-letter mapping) that went unnoticed during code review.
The complexity scores for the multi-model dispatch function in \texttt{main.py} tracked
the refactoring process: the cognitive complexity score dropped from 18 to 9 after
extracting four helper functions, and the Compare page confirmed the improvement before
the change was committed.

Applied to the ten LOPO evaluation repositories, the platform processed 3,875 Python
functions without error and surfaced complexity rankings consistent with manual inspection
of the highest-scoring functions in each repository. In \textit{sqlalchemy/sqlalchemy},
the five highest-scoring functions were all in the ORM query compiler, the part of the codebase that, by general reputation, is the most architecturally complex. That
qualitative match does not constitute validation, but it is evidence that the output is
not random.

To provide at least a directional precision estimate for one deterministic module, the
dead code detector was applied to twelve Python files drawn from three of the LOPO
evaluation repositories (\textit{pallets/flask}, \textit{psf/requests},
\textit{bottlepy/bottle}) and each finding was checked manually by the thesis author
against the AST and documentation to determine whether the flagged symbol was genuinely
unreachable, unreferenced, or empty. Of 31 findings produced, 26 were confirmed as
genuine dead code (estimated precision $\approx 0.84$). Four false positives arose from
module-level names exported via \texttt{\_\_all\_\_} that the detector does not currently
inspect, and one from a method intentionally left as a placeholder stub. No ground-truth
positive set was constructed, so recall cannot be reported. This is a single-author
spot check, not a controlled evaluation; the precision estimate carries no statistical
guarantee and should be read as ``no obvious systematic error was found,'' not as a
validated accuracy claim. A controlled evaluation with two independent annotators and a
complete positive census would be needed to establish confident precision and recall
figures.

These observations are indicative, not rigorous. They are consistent with correct
operation and sensible output; they do not establish that developers act differently
because of the findings.

\subsection{Threats to validity}

No evaluation is free of threats, and acknowledging them is essential to interpreting
results correctly. Following the framework of Wohlin et al.~\cite{wohlin2012}, four
categories of threats are relevant here.

\textit{Internal validity} was the subject of the audit correction described in
Section~\ref{ssec:audit}. The label leakage defect in the original complexity model
was a serious internal validity threat because it made the model's performance appear
determined by genuine learning when it was in fact determined by a tautological
relationship between features and target. After correction, the primary remaining
internal validity concern is that all three classifiers share training data from a
relatively narrow set of well-maintained open-source projects, which may not represent
the variation a real developer encounters.

\textit{External validity} is the most significant ongoing concern. The models are
trained exclusively on Python code from popular, well-maintained repositories.
Generalisation to other domains (scientific computing, embedded firmware, financial applications, legacy enterprise systems) is entirely untested. The LOPO results give
some comfort for complexity ranking within the Python open-source ecosystem, but they
say nothing about transfer to other programming languages or organisational codebases.

A related concern is dataset scale. After expanding the corpora by mining additional
samples from the locally cached OSS repositories, the security, complexity, bug, and
pattern datasets contain 4,286, 3,875, 1,200, and 1,153 samples respectively, totalling around 10,500 samples across four tasks. The security and complexity datasets are now
substantially larger than in the initial version of this work (1,286 and 875 samples
respectively), and the improved metrics reflect this: security multi-seed AUC rose from
0.834 to 0.963, and complexity RMSE dropped from 49.5 to 28.2. The bug and pattern
datasets remain at their original sizes, and their metrics are discussed in context below. This is on the smaller end for ML research,
though it is well within the range found in comparable published work. Li~et~al.~(2017)
train their vulnerability classifier on roughly 1,000 functions; D'Ambros~et~al.~(2010)
evaluate bug prediction across three projects totalling fewer than 2,000 releases;
Palomba~et~al.~(2018) use under 1,500 manually validated code smell instances. The
LOPO evaluation protocol partially compensates for the limited scale: by constructing
ten independent train-test splits and measuring generalisation on completely unseen
repositories, it tests robustness more stringently than a larger dataset evaluated with
a single random split. Nevertheless, expanding these datasets, particularly the complexity and security corpora where sampling additional functions from a broader set of repositories is straightforward, is the most tractable path to reducing this threat.

\textit{Construct validity} is relevant for the pattern recognition task, where the
oracle is constructed from a combination of PyNose output and metric thresholds. One
specific threat is that the updated oracle uses function length and parameter count to
assign labels, the same quantities that appear in the input feature vector. After removing the 14 leaky features and retraining on 12 structural AST features, the
debiased CV F1 is $0.657 \pm 0.016$, a quantified $+0.082$ drop from the leaky
result, confirmed by Wilcoxon signed-rank test ($p = 0.031$). Cohen's kappa of
$\kappa = 0.168$ (binary) between the debiased model's oracle labels and an independent
metric-threshold oracle confirms that the label agreement is slight, consistent with
the residual construct-validity concern even after debiasing. A further concern is that different threshold choices produce
different labels, making the oracle itself uncertain: the model may be learning one
particular threshold calibration rather than an underlying construct.

A separate concern affects the security LOPO evaluation. After the initial dataset of
1,286 samples proved too small, additional samples were mined from the same open-source
repositories that appear in the LOPO fold assignments. Repositories used to expand
the training corpus should not appear as held-out LOPO projects; to the extent that
they do, the LOPO AUC of 0.494 may be slightly optimistic about cross-project
generalisation. Resolving this fully would require re-mining from a disjoint set of
repositories, which is left as future work.

\textit{Conclusion validity} is a concern for the bug prediction results in particular.
The temporal split evaluation involves a single train-test boundary; repeating the
evaluation at multiple cut points across the commit timeline would produce a more stable
estimate of the temporal AUC. This is left as future work.

\subsection{Positioning relative to existing tools}

The comparative evaluation in Section~\ref{sec:market-comparison} makes the positioning
concrete. Table~\ref{tab:coverage} shows that no open-source tool without dedicated
infrastructure covers more than five of the twelve analysis dimensions IntelliCode Review
addresses; most cover one or two. Among tools that operate without a server, Pylint is
the closest competitor, providing partial coverage across six categories but lacking
clone detection, documentation quality, performance analysis, and dependency scanning
entirely. Bandit and Semgrep each address security exclusively. Radon and Vulture are
both single-dimension tools.

On the security detection task, IntelliCode Review achieves AUC 0.963 versus Bandit's
0.500 and Semgrep's 0.503 on the same evaluation set
(Table~\ref{tab:market-security}). The label-alignment caveat noted in
Section~\ref{sec:market-comparison} applies: Bandit performs well on codebases with
explicit insecure idioms and should be treated as a complement rather than a substitute.
On complexity, the XGBoost model produces a Spearman $\rho$ of 0.819 against cognitive
complexity ground truth, compared to 0.724 for Radon's McCabe CC and 0.836 for the
LOC naive baseline (Table~\ref{tab:market-complexity}).

SonarQube Community Edition is the only existing tool that approaches comparable
dimension coverage. The distinction that matters most in practice is deployment: SonarQube
requires infrastructure that most individual developers cannot or will not stand up.
IntelliCode Review makes the same breadth of analysis available to a developer who has
Python and a browser.

Compared to commercial tools such as Snyk or DeepCode (now part of Snyk), IntelliCode
Review is fully open: all training scripts, feature extraction code, and model
checkpoints are included in the repository and can be inspected, retrained, or replaced.
This transparency is important for research reproducibility and for practitioners who
need to understand why a finding was raised. The cost of transparency is narrower
training data and the absence of the large proprietary vulnerability datasets that
commercial tools draw on. For the use case this thesis targets, a research prototype that demonstrates how far a self-contained ML-augmented analysis tool can go with public data and open-source components, this is an acceptable and honest trade-off.

\section{Preliminary interface sanity check}
\label{ssec:peer-eval}
%% ============================================================

This section reports a preliminary interface sanity check with four graduate students,
not a usability study. The distinction matters. A usability study requires a
representative user sample, realistic task scenarios drawn from participants' own work,
pre-registered hypotheses, and statistical power adequate to detect meaningful
differences. None of those conditions are met here. What follows is an initial check
for gross output-interpretation failures and obvious workflow friction before the more
rigorous study described in Section~\ref{sec:future}.

Four graduate students in computer science each submitted a Python code sample, read the
resulting IntelliCode Review report, and completed three tasks: identify the key issues,
interpret the quality score, and assess at least one suggested fix. All four completed
every task without assistance, and no participant reported difficulty on any of the four
questionnaire criteria (all responses on a 5-point Likert scale were at or above 4).
That is the extent of the claim: the interface did not produce obvious navigation
failures for this group. No quantitative means are reported, because with $n = 4$ the
numbers carry no statistical information.

Actionable feedback came from two participants who could not interpret the probabilistic
bug risk score without additional context; this motivated the \texttt{reliability\_context}
field in the bug predictor API response, which now surfaces the three evaluation-protocol
AUC values alongside the probability estimate.

The limitations are severe. The sample shares a research background with the system's
developer, introducing experimenter-familiarity bias. The scenario did not involve
participants' own code or real development pressure. A controlled study with professional
developers, realistic tasks, and pre-registered outcome metrics (time-to-first-issue,
issues acted on, pre/post static-analysis score) is the required next step before
any usability claims can be made.

\section{Summary of contributions}
\label{sec:contributions}

The central question driving this work was practical: can a single tool give a developer
comprehensive, honest code quality feedback in under two seconds, without requiring a
server rack or a DevOps budget? Yes, the platform does that. But that is the secondary
result. The primary result is what the evaluation reveals about what the models actually
do when you stop measuring them on held-out splits from their own training distribution
and start asking how they behave on code they have never seen, or on commits that
happened after the training window closed. Those results are less flattering, and they
are the point.

IntelliCode Review brings all twelve analysis dimensions into a single web application
that any developer can deploy locally in a few minutes: four driven by trained ML models, eight by deterministic AST-based modules. Existing open-source tools specialise
in one or two dimensions (Bandit for security, Radon for complexity, Pylint for style),
and SonarQube, which is comparably broad, requires a server, a database, and an
administrative burden that makes it inaccessible for individual developers on personal
or early-stage projects.

The ML results are uneven — full details are in Chapter~\ref{ch:eval}. The short
version: complexity ranking transfers across nine of ten held-out repositories
($\rho = 0.819$, SonarQube calibration $r = 0.980$) after fixing a label-leakage defect
that had inflated the original result to $R^2 = 1.000$. Security detection
in-distribution AUC ($0.963 \pm 0.012$) collapses to chance under LOPO
($0.494 \pm 0.189$); the ablation pinpoints feature scope as the binding constraint,
not label noise. Bug prediction is below chance under a temporal split ($0.460$); the
random-split figure of $0.676$ is inflated by temporal leakage. The pattern classifier's
debiased CV F1 of $0.657 \pm 0.016$ (Cohen's $\kappa = 0.168$) reflects near-chance
agreement with any ground truth outside its own oracle; the production interface now
shows raw structural AST feature values rather than class labels.

The evaluation protocol matters independently of the model results. The claim it
supports is stronger than it might appear: \textit{no published ML-based code quality
system should be trusted on in-distribution AUC alone}. The tools and results of this
thesis demonstrate that concretely. The complexity model's original $R^2 = 1.000$ was
caught by a calibration check, not by the held-out split. The security model's
in-distribution AUC of $0.963$ collapsed to $0.494$ under LOPO. The bug predictor's
$0.676$ became $0.460$ under temporal evaluation. In each case, the standard evaluation
would have been misleading. The instruments in this thesis are designed to make those
collapses visible before deployment rather than after.

An integrated evaluation protocol comprising ten diagnostic checks (Section~\ref{sec:novel-contributions}) was developed to address these gaps: PIFF and SAEF for generalisation diagnostics, CACM for calibration monitoring, TDCP for temporal robustness, LAMV and PLF for leakage detection, and ACT, FTAX, and SIG for actionability and failure analysis. Adding five-seed variance, LOPO benchmarks, walk-forward temporal CV, conformal intervals, and SonarQube calibration, these give a picture of deployable performance that a single split cannot provide.

The protocol is a concrete, runnable artefact. \texttt{backend/evaluation/} contains one Python module per instrument, each with a command-line interface; \texttt{backend/run\_novel\_contributions.py} runs the full suite via \texttt{--only PIFF SAEF LAMV} flags and persists JSON results to \texttt{evaluation/results/}. Any researcher with a dataset conforming to the documented schema can substitute their own trained models and run the full protocol without modifying IntelliCode Review itself.

The multi-language extension demonstrates that the deterministic analysis infrastructure
can be extended to new languages without retraining the ML models. Tree-sitter parsers
for JavaScript, TypeScript, and Java expose the same structural metrics that feed into
the security and complexity modules, and language-specific pattern rules provide
vulnerability coverage for the three most common web development languages alongside
Python. This matters because real codebases rarely consist of a single language, and
a tool that silently ignores non-Python files will miss a significant fraction of the
risk surface. A NIST SARD-inspired evaluation of the JavaScript rules against 52 labelled snippets
(26 vulnerable, 26 clean, eight CWE categories) is reported in
Section~\ref{ssec:multi-language-adapter} and Table~\ref{tab:js-security-eval}:
micro precision 1.000, recall 0.885, F1 0.939, with zero false positives on clean
snippets. Java rules remain unevaluated against labelled data and should be treated
as structural hints rather than validated findings.

The developer interface is designed around one observation: a finding that a developer
cannot act on in the next five minutes will usually be ignored. The fourteen-tab Review
Detail page, the Fix Guide with copy-ready remediation code, the Compare page, and the
SSE streaming endpoint that pushes partial results to the browser as each model finishes
all reflect that constraint. The GitHub Actions workflow
(\texttt{.github/workflows/intellicode-review.yml}) extends the same idea into
continuous integration: every pull request that touches Python, JavaScript, TypeScript, or Java files triggers a
self-contained analysis run, and findings appear as inline review comments on the
changed lines. Whether this actually changes developer behaviour, whether developers read and act on the findings rather than dismissing them as noise, is an empirical question that a controlled study with professional developers would need to answer.
The peer-based evaluation (Section~\ref{ssec:peer-eval}) provides preliminary
evidence that the output is interpretable and the Fix Guide is useful; it does not
establish whether developers act on findings differently in a real workflow.

\section{Answers to research questions}
\label{sec:rq-answers}

\textbf{RQ1: Can a single platform provide comprehensive, multi-dimensional code
quality analysis that is both deeper than existing static tools and practical for
everyday developer use?}

Yes, within the stated scope.
Table~\ref{tab:coverage} in Section~\ref{sec:market-comparison} documents the coverage
gap directly: IntelliCode Review addresses all twelve analysis dimensions; Pylint,
the next most comprehensive open-source tool, covers five. SonarQube Community Edition
covers eleven but requires a dedicated server and database that most individual
developers will not set up. The 100\% pass rate on 24 end-to-end API test cases confirms
the integrated pipeline is mechanically correct. The Fix Guide, SSE streaming, and
Compare page address workflow friction that rule-based tools ignore entirely.

The honest caveat is that coverage and depth are not the same thing. Bug prediction and
pattern recognition have model limitations that make them unreliable as standalone
predictors. The platform's value comes from surfacing many signals simultaneously, not
from being exceptionally deep in any one.

\textbf{RQ2: Can machine learning models trained on real open-source data provide
reliable code quality estimates that are competitive with the current literature
and honest about their generalisation limits?}

Partially, and the gap between in-distribution and cross-project results is the most
important finding. The complexity model transfers reasonably well (LOPO $\rho = 0.795$
across ten projects). The security model does not: in-distribution AUC $0.963$ becomes
chance-level $0.494$ under LOPO. The bug predictor is below chance under temporal
evaluation ($0.460$). The pattern classifier's debiased F1 of $0.657 \pm 0.016$
reflects genuine structural signal after removing circular label supervision, but
Cohen's $\kappa = 0.168$ indicates near-chance agreement with any independent label
source. Full numerical details are in Section~\ref{sec:model-results}.

\textbf{RQ3: Can the system be delivered as a practical, self-contained tool
with response times fast enough to support interactive developer workflows?}

Yes, within the stated constraints. All inputs up to 1000 source lines of code are
analysed in under two seconds on a development machine without GPU acceleration. The
\texttt{/analyze/quick} endpoint stays below 400\,ms for 100-line files, making it
viable for CI integration. SSE streaming delivers the first model results within 200\,ms
of submission, so the interface remains responsive during full-pipeline analysis.

The dependency footprint is real: Python with Uvicorn, FastAPI, PyTorch, XGBoost, and
scikit-learn, plus Node.js for the frontend. That is manageable with a standard
development environment. What is not required is any cloud service, database, or
administrative setup: no accounts, no network calls at analysis time, no persistent storage beyond the browser's localStorage. For a developer working on a local or
early-stage project, that is the distinction that matters.

\section{Implications for practitioners}
\label{sec:implications}

The evaluation results have practical consequences. The interesting question is not
whether the models fail (they clearly do in cross-project settings) but \textit{why} they fail and what that implies for which components a practitioner should actually rely on.

Eight of the twelve analysis dimensions are rule-based AST traversals: clone detection,
dead code, refactoring suggestions, technical debt, documentation, performance patterns,
dependency coupling, and readability. These components have no trainable parameters and
no distribution shift problem. A developer deploying IntelliCode Review on a codebase
the models have never seen should weight these findings heavily and treat ML outputs
as soft signals that supplement, not replace, the deterministic findings.

Among the ML components, the XGBoost complexity regressor transfers best. It achieves
Spearman $\rho = 0.819$ in-distribution and $0.795$ under LOPO, a relatively small gap.
Complexity is also physically interpretable: a cognitively complex function is complex
regardless of the project it came from. For practitioners who want one ML number to
trust, this is it. The model exposes its top-5 feature importances in the API response
so callers can see which structural features drove the score on a given file.

The bug predictor cannot reliably identify which functions will introduce bugs in the
future. The temporal AUC of 0.460 (below chance) is clear on that point. What it can
do is score structural complexity risk: files with high cognitive complexity, many
exception blocks, and long functions score higher, which is a useful proxy for where to
focus a code review. The system surfaces this distinction in the API response via a
\texttt{reliability\_context} field that reports all three evaluation-protocol AUC
values (in-distribution: 0.676, temporal CV: 0.544, LOPO: 0.567) alongside a plain-English
recommended-use note.

The security model's LOPO AUC of 0.494 means it only reliably detects vulnerability
types present in its training data. For repositories that use infrastructure patterns,
scientific computing idioms, or domain-specific frameworks outside the training corpus,
the rule-based scanner is the reliable component and the RF+CNN ensemble should be
treated as supplemental. Any vulnerability finding confirmed by both the rule-based
scanner and the ML ensemble can be treated as high confidence; ML-only findings should
prompt human verification.

The primary bottleneck is not architecture but label quality and training data scope.
Anyone retraining these models should replace SZZ commit-message heuristics with
closed bug reports from issue trackers, where the ground truth is less noisy. Adding
a few dozen labelled examples from the target codebase, even a small project-specific fine-tuning set, would likely do more for cross-project performance than any architectural change. The LAMV leakage check and LOPO protocol from this thesis's evaluation framework
are designed to be reused: all scripts in \texttt{backend/evaluation/} run on any
dataset that conforms to the documented schema, so verifying that a retrained model
actually improves rather than just overfits is straightforward.

\section{Future work}
\label{sec:future}

Four directions stand out as the most impactful next steps. They are listed roughly in order of expected payoff.

\textbf{Dataset expansion with representation learning.} The clearest path to improving LOPO performance is not a better model; it is better training data. The
current dataset covers 33 repositories across five Python domains; a systematic
expansion to 100+ repositories spanning scientific computing, systems programming,
financial applications, and embedded firmware would test whether the LOPO collapse
is a Python-web-framework artifact or a fundamental limit of static-metric
representations. Alongside this, replacing the hand-crafted 15-dimensional feature
vector with a learned code representation (e.g., CodeBERT, GraphCodeBERT, or
AST-based GNNs) would eliminate the manual feature engineering bottleneck entirely.
A representation learner pre-trained on 100 million lines of code can transfer far
more robustly across project boundaries than a set of scalar metrics. The experiment
is straightforward: swap the XGBoost input with the \texttt{[CLS]} embedding from a
frozen CodeBERT encoder and re-run the LOPO benchmark. If LOPO Spearman rises from
0.795 toward 0.85+, the current feature engineering is the bottleneck. If it does
not move, the label quality problem dominates.

\textbf{A pre-registered real-world deployment study.} The peer-based evaluation in
Section~\ref{ssec:peer-eval} established that the interface is interpretable to
graduate students working on controlled exercises. That is a necessary condition for
usefulness, not a sufficient one. The research question that matters, whether IntelliCode Review changes what developers fix before a pull request merges, requires a controlled
study with professional developers, their own production codebases, randomised tool
assignment (IntelliCode Review vs. single-purpose linter), and pre-registered outcome
metrics: issues flagged per session, issues acted on, time-to-fix, and pre-to-post
static analysis score improvement. Pre-registration is particularly important here
because the space of outcomes is large and exploratory analysis of tool usage logs
will find patterns regardless of whether the tool had a causal effect.

\textbf{Temporal AUC stability.} The below-chance temporal AUC of 0.460 is measured
at one 80/20 split boundary. Running five or more walk-forward evaluations at
different cut points would establish whether anti-learning is stable across the commit
timeline or localised to a particular period where training and test data diverge most.
If stable, it confirms a genuine limit of static-metric bug prediction. If variable,
it identifies the historical window where the model fails, which is itself useful for
deciding where keyword-label retraining efforts should focus.

\textbf{The pattern oracle problem and interface restructuring.} The pattern model's
F1 of $0.657$ is measured against a labelling oracle built from PyNose and metric
thresholds that partially overlap the input feature space. A stronger oracle, based on
maintainer-accepted refactoring commits across a large corpus, would establish an upper
bound on what metric-free pattern detection can achieve. Until that oracle exists, reporting four-class label probabilities as the primary
display overstates the classifier's diagnostic precision ($\kappa = 0.168$). The production
interface was updated in response: the pattern tab now displays the twelve raw structural
AST feature scores (nesting depth, function count, maximum body length, import count, and
related metrics) as metric cards with threshold-based highlighting, rather than a class
label and confidence bars. A developer who sees that their function has a nesting depth
of 6 and a body length of 72 lines can act on those facts directly without having to
accept an uncertain label assignment.

Two engineering extensions are less research-driven but would substantially expand the
tool's applicability. Full multi-language ML coverage is straightforward given the
tree-sitter infrastructure already in place: the language adapters for JavaScript,
TypeScript, and Java extract AST features that mirror the Python feature vector, so
training language-specific models requires data collection, not architectural change.
Replacing browser localStorage with a lightweight server-side store would turn
IntelliCode Review from a single-user tool into a team platform, with shared rule
sets and longitudinal tracking across release cycles.

\section{Closing remarks}
\label{sec:closing}

The central bet that IntelliCode Review makes is that breadth of coverage, honest
evaluation, and actionable output matter more to a developer than any single model's
predictive accuracy in isolation. A security scanner that flags SQL injection in an
otherwise simple function is useful even when the complexity model gives that function
a perfect grade. An analysis that reports not just a score but a parameterised query
template the developer can copy immediately is useful in a way that a score alone is
not. A system that surfaces results in under two seconds, with the first tab populated
in under 200 milliseconds, is used differently than one that requires a separate
analysis run.

The evaluation results show real but bounded capability. The complexity model ranks
functions correctly across most Python projects. The security model catches the
vulnerability patterns in its training data and misses patterns outside that set. The
bug predictor's temporal-split AUC of 0.460 and the pattern classifier's circular
labelling oracle both constrain what can honestly be claimed for those two modules. The
temporal split and LOPO results are not problems to be buried in an appendix; they are
the honest answer to the question of what this system actually does under conditions
that approximate deployment.

This thesis is an argument for a particular way of doing applied ML research in software
engineering: use real data, build the evaluation to stress-test results rather than
confirm them, and report what you find. The system is the means; the evaluation is the
contribution. IntelliCode Review does not solve code quality analysis. It shows where the current generation of feature-based ML approaches works, where it breaks, and why.
One developer, working from public datasets and open-source libraries, can build something
broader and more honest than what most teams currently run. Whether that is impressive
or sobering depends on what you expected going in.

\appendix

\chapter{Frontend Component Reference}
\label{app:frontend}

This appendix documents the React components that implement each page of the
IntelliCode Review interface. Component-level details are separated from the main
implementation chapter to keep the architectural discussion focused on design
decisions rather than code inventory.

\section{Navigation, Routing, and Authentication}

The React application uses React Router~v6 for client-side routing. The sidebar
navigation (\texttt{AppNavigation.tsx}) links to ten primary pages visible to all
users (Dashboard, Submit, Batch, Repositories, PR Review, Reviews, Compare, Analytics,
Rules, and Settings), with Admin conditionally shown to privileged users. A \texttt{ProtectedRoute} component wraps authenticated routes; users who
have not started a session are redirected to the Login page. The navigation bar
includes a live API status indicator that polls \texttt{GET /health} every 15 seconds
(3-second abort timeout), displaying a pulsing green dot when the backend is reachable
or a red dot when it is offline.

\section{Code Submission Page}

The Submit page (\texttt{Submit.tsx}) provides an editor chrome with line numbers, a
Copy button that writes the current buffer to the clipboard (showing a checkmark for
2 seconds), and a submit button labelled \textit{Ctrl+Enter} on desktop. A
\texttt{useEffect} hook registers a \texttt{keydown} listener for \texttt{Ctrl+Enter}
that triggers analysis when the editor is active. Batch repository scans navigate to
the Reviews list on completion. On submission, \texttt{api.ts}'s \texttt{analyzeCode()}
posts to \texttt{/analyze}, saves the result via \texttt{reviewHistory.ts}, and navigates
to the Review Detail page.

\section{Analysis History and Sorting}

The \texttt{reviewHistory.ts} service manages past analysis results in
\texttt{localStorage} under the key \texttt{intellcode\_history}. Each entry includes
a UUID, timestamp, filename, overall score, status, and the full API response. The
Reviews page renders this list with a multi-field sort control (Newest First, Lowest
Score, Most Issues, Highest Severity) and text filtering.

\section{Compare Page}

The Compare page (\texttt{Compare.tsx}) presents two side-by-side code editors, each
with an independent ``Analyze'' trigger. Selecting ``Run Comparison'' calls both
\texttt{/analyze} endpoints in parallel, then renders a diff-style metrics table
highlighting which submission scored better in each dimension. A ``Reset Samples''
button restores both slots to built-in example snippets.

\section{Dashboard and Analytics}

The Dashboard (\texttt{Dashboard.tsx}) displays summary statistics, a quality trend
sparkline (Recharts \texttt{AreaChart}), recent reviews, and -- when two or more
distinct files have been analysed -- a leaderboard showing the five highest-scoring
files and five lowest-scoring files. The dashboard updates via \texttt{storage}
events and a 5-second polling interval.

The Analytics page (\texttt{Analytics.tsx}) renders trend charts (score over time,
status distribution, per-category distributions, score histogram) from localStorage.
A ``File Quality Progress'' section shows per-file \texttt{LineChart}s for up to five
files with at least two analyses, colour-coded by improvement or decline.

\section{Admin Page}

The Admin page (\texttt{Admin.tsx}) provides a model status panel (ready / unavailable /
not loaded for each of the twelve models), four statistics cards (total cached analyses,
average score, total security findings, models ready) sourced from \texttt{GET /stats},
and a ``Clear Cache'' button that calls \texttt{POST /cache/clear}.

\section{Settings Page}

The Settings page (\texttt{Settings.tsx}) organises preferences into tabs: Profile,
Notifications, Appearance, and Integrations. The Notifications tab reads
\texttt{Notification.permission} on mount, displays the current permission status
(Enabled / Blocked / Not set), and provides an ``Enable'' button that calls
\texttt{Notification.requestPermission()} and fires a test notification. Notification
permissions are requested lazily on user action rather than at page load, following
browser best-practice guidelines that gate permission prompts behind an explicit user
gesture.

\section{Pull Request Review Page}

The PR Review page (\texttt{PRReview.tsx}) provides a two-step GitHub pull request
analysis workflow. In the first step the developer pastes a GitHub PR URL and triggers
analysis; the frontend calls \texttt{POST /github/analyze-pr} with
\texttt{post\_comments: false}, receives per-file results, and renders them using the
same Security and complexity components as the single-file Review Detail page. A
collapsible preview panel shows a markdown summary of what would be posted to GitHub,
including the number of inline comments and which findings are excluded by the
confidence threshold. In the second step the developer confirms by clicking a distinct
``Post Comments to PR'' button, which calls the same endpoint with
\texttt{post\_comments: true}; comment URLs returned by the API are then displayed as
clickable links. The GitHub personal access token is read from localStorage via
\texttt{github.ts} so no manual token entry is required after the initial OAuth
connection. If the token is absent, a banner directs the user to the Settings page.
Stable React keys on all list items (derived from finding title and line number rather
than array index) prevent reconciliation artefacts when the results list changes between
the analyze and post steps.

\section{Rules and Repositories Persistence}

The Rules page allows users to define, enable/disable, and configure custom quality
rules with an embedded code preview panel. The preview panel re-analyses the typed
snippet in real time, giving immediate feedback on whether a proposed rule would flag
the intended pattern. Rules state is persisted in localStorage across sessions. The
Repositories page persists repository configurations (name, URL, branch, language) and
initiates batch scans across sixteen recognised programming languages.
